# Reactivity bench — run 001

> **Date:** 2026-05-17
> **Harness:** `nmp-testing/bin/reactivity-bench` (initial implementation)
> **Design under test:** `docs/design/reactivity.md` rev 0
> **Outcome:** Design partially validated; two refinements identified before Phase 1 lock-in.

---

## Headline

**Reverse-index direction validated. Composite-dependency model required. Delta budget and memory budget both reframed.**

The reverse-index lookup and per-view recompute paths have 100×–1000× headroom against their gates. The current dependency model produces excessive false wakes. The delta-volume gate was set as an absolute and should have been per-view. The memory gate assumed all events resident and should have been working-set only.

## What the data says

### Lookup and recompute are non-issues

The reverse-index direction (`docs/design/reactivity.md` §3) is validated. Both critical paths have so much headroom that further optimization there would be premature. **Lock these in.**

### Naive union of dependency buckets is the design smell

In the v0 model, a timeline with `kinds=[1,6,7]` and `authors=[A,B,C]` registers under `by_kind[1,6,7]` ∪ `by_author[A,B,C]`. When a kind:1 arrives from author Z (not in the view's author set), the view wakes anyway because of the `by_kind` registration. Recompute correctly returns `None`, but the wake itself is waste.

At quiet_idle (1 event/sec, 10 views), 98% of wakes return `None`. At following_timeline_scroll (1k authors × 3 kinds × 1 view, 200 events/sec), 49% return `None`.

The fix: **conjunctive dependencies** — register under the most specific composite key supported. See §4 below for the refined design.

### Delta-volume gate was wrong

The 1,000 deltas/sec absolute gate doesn't model fan-out. profile_fanout hit 3,379.56/sec because 50 views legitimately need to be told the same author changed — that's structural, not noise.

**Refined gate: ≤ 60 deltas/sec/view.** This matches the 60Hz flush and is what within-view coalescing produces naturally.

Under the refined gate:

- hashtag_firehose: 2,000/sec on 1 view = 2,000/view/sec — still fails, real coalescing problem.
- profile_fanout: 3,379.56/sec on 50 views = ~67.6/view/sec — barely fails, coalescing closes it.

Both fail to the same fix: **within-view delta coalescing at flush time** (§5 below).

### Memory gate was wrong

The 100 MB gate at 1M events assumed all events resident in memory. This is the anti-pattern the spec already calls out: the storage backend (LMDB / SQLite / IndexedDB) holds events; the actor keeps a bounded working set hot.

**Refined gate: working-set memory ≤ 100 MB**, measured at 100 active views and ≤ 10k hot events. Total cached events on disk is unbounded (or capped by the storage backend's quota).

The reverse index does not need event bodies — only attributes (kind, author, e/p/d tags). It can index every event on disk without resident bodies. Projection caches (author_display, reaction_summary, etc.) are LRU-bounded by referenced-view count.

### Allocation gate is not proven

The harness reported "allocations: unmeasured." Plumb a counting allocator (`dhat` or jemalloc stats) into the harness so the zero-allocation-per-event invariant is verifiable, not aspirational.

## Design refinements adopted in `reactivity.md`

### §3 (Dependency tracking) — composite keys are primary

Replace the v0 model "register under each axis independently" with: register under the **most specific composite key** the view's dependencies support.

| View shape | Primary index |
|---|---|
| `kinds + authors` | `by_kind_author[(k, a)]` cartesian product |
| `kinds + e-tag refs` | `by_kind_e_tag[(k, e)]` |
| `kinds + p-tag refs` | `by_kind_p_tag[(k, p)]` |
| `kinds + d-tag refs` (parameterized replaceable) | `by_kind_author_d[(k, a, d)]` |
| `kinds` only (broad) | `by_kind[k]` — broad-cost flag |
| `authors` only | `by_author[a]` — broad-cost flag |
| no constraint (catch-all) | `catch_all` — explicit guardrail warning |

Views in the "broad-cost flag" tier produce a `nmp-guardrails` warning in debug builds; they are legal but expected to be rare (search, hashtag scan).

On insert, generate the event's tuple signature — every (kind, axis-value) pair the event implies — and look up each. Union the small resulting sets. No more "kind:1 wakes everyone subscribed to kind:1."

### §5 (`ViewDelta` catalog) — within-view coalescing at flush

Add a coalescing pass to `DeltaBuffer::flush()`:

```rust
fn flush(&mut self) -> Vec<ViewDelta> {
    self.deltas.sort_by_key(|d| d.view_id());
    let mut out = Vec::new();
    for (view_id, group) in self.deltas.chunk_by_view_id() {
        out.extend(coalesce(view_id, group));
    }
    out
}

fn coalesce(view_id: ViewId, deltas: Vec<ViewDelta>) -> Vec<ViewDelta> {
    match view_kind(view_id) {
        ViewKind::Timeline => coalesce_timeline(deltas),
        ViewKind::Reactions => coalesce_reactions(deltas),
        ViewKind::Conversation => coalesce_conversation(deltas),
        // ...
    }
}
```

Per-view-kind coalescing rules (illustrative — full enumeration in `view-catalog.md` Phase 1):

- **Timeline:** consecutive `Inserted { at, items }` at adjacent positions merge into one `Inserted { at, items: combined }`. Multiple `Updated { id, item }` to different ids stay as one `UpdatedMany` if all share an `AuthorPatch`. `Removed { ids }` accumulate.
- **Reactions:** N `EmojiAdjusted { emoji, delta }` for the same emoji collapse to one with summed delta. Different emojis stay separate.
- **Conversation:** consecutive `Appended { messages }` merge.
- **Profile:** only `Replaced` exists; later supersedes earlier within tick.

Coalescing happens **after** all view callbacks return for the tick. The actor doesn't need to know which deltas might coalesce; the buffer handles it.

### §7 (Scheduling and batching) — flush triggers

Unchanged: time-based (16 ms / ~60 Hz), size-based (256 deltas pre-coalesce, or post-coalesce equivalent), or forced (account switch, view open).

### EventStore working-set discipline

Add to `reactivity.md` §3 and `product-spec.md` §7.1: the EventStore holds a **bounded hot working set** in memory; cold events live in the storage backend. The reverse index indexes both. Lookups against cold events return their ids and attributes immediately; loading event bodies for delta construction happens lazily and synchronously via the backend.

Working-set policy:
- Hot: events referenced by any open view's claim set + recent N events (N configurable, default 5,000).
- Cold: everything else, on disk only.
- Eviction: LRU among hot events not currently claimed.

This makes the "1M events at 130 MB" finding a non-finding: total cached events is unbounded; working-set is bounded.

## Items not actioned (yet)

- **Cross-view delta coalescing.** A kind:0 arrival hits 50 views; each produces one `UpdatedMany` delta keyed by its view id. Within-view coalescing reduces per-view delta count, but cross-view count is bounded by interested-view count. Open question: is there a delivery format that says "this AuthorPatch applies to view ids [V1..V50] of kind Timeline" with one entry instead of 50? Breaks the per-view-id contract; defer to a separate study if profile fan-out persists as a problem after the refinements land.
- **Storage-tier interaction with reverse-index updates.** When a cold event is loaded for delta construction, we don't need to add it to the working set unless it'll be re-referenced. Define the policy explicitly in the EventStore design.

## What this run validates about the process

The harness ran before any view-kind code shipped. It found two budget bugs and one design smell that argument alone wouldn't have surfaced. The reverse-index core is preserved; two patches close the gaps; gates are refined to be actually measurable.

**Pattern to enshrine:** every design doc has gates; gates run on the harness; failures revise the design *before* implementation. Pre-implementation measurement is cheaper than post-implementation rework.
