# Codex Review — ec1e205

**Subject**: feat(subs): dead-relay exclusion before apply_selection
**Batch**: HB60-HB66
**Date**: 2026-05-18
**Verdict**: NEEDS-FIX

## Summary

Adds `dead_relays: HashSet<RelayUrl>` to `SubscriptionLifecycle`, implements
`mark_relay_dead()` / `mark_relay_alive()`, filters dead relays from the compiled
plan before `apply_selection`, and introduces `CompileTrigger::RelayHealthChanged`
(A12). Three new tests added in `subs/mod.rs`.

## Rubric

### Correctness

1. **Wire-diff relay-blindness** (HIGH): the same bug as 8fdc2ff: `plan_diff` builds
   global `BTreeSet<sub_id>` sets rather than `(relay_url, sub_id)` sets. When a
   relay is marked dead, the `.retain()` removes it from `per_relay`, but if the
   surviving relay carries the same filter hash, the diff will not emit a `CLOSE` for
   the dead relay. When marked alive again, the revived relay may not get a `REQ`.
   Fix: scope diff keys to `(relay_url, sub_id)`. See `wire.rs:56`.

2. **Test gap** (MEDIUM): the three new dead-relay tests assert `current_plan` shape
   (relay count) but do not assert emitted `WireFrame`s. This misses the bug above.
   Add tests that assert: (a) marking a relay dead in a mixed plan produces
   `WireFrame::Close { relay_url: "wss://dead.example" }`, and (b) marking it alive
   again produces `WireFrame::Req` for the resurrected relay.

3. **Generic trigger handling**: `RelayHealthChanged` will cause a recompile via the
   normal `drain_tick()` path (any non-empty inbox). No dedicated handler is needed
   for correctness, but `drain_tick()` call sites are tests-only — the actor tick
   path is not yet wired for this trigger in production.

4. **API asymmetry** (non-blocking): `mark_relay_dead(url: RelayUrl)` takes ownership
   while `mark_relay_alive(&RelayUrl)` borrows. Not a correctness issue at current
   call sites.

5. **Budget isolation** (non-blocking): `set_selection_budget(usize::MAX, usize::MAX)`
   in the new tests is acceptable for isolating dead-relay filtering from selection
   pruning, but absence of WireFrame assertions means the test doesn't verify the
   purpose of the filtering step.

### Tests

Three new tests pass. All assert `current_plan.per_relay.len()` but none assert
`WireFrame::Close` or `WireFrame::Req` for the dead/alive relay transitions.

### Architecture Fit

Dead-relay exclusion before `apply_selection` is the correct pipeline position. Using
`.retain()` on the compiled plan rather than modifying the compiler is architecturally
clean. The `RelayHealthChanged` trigger variant fits the A-series naming.

### Follow-Ups

- Fix `plan_diff` to scope by `(relay_url, sub_id)` (shared with 5634896 and 8fdc2ff).
- Add WireFrame-asserting tests for dead/alive relay transitions.
- Wire `RelayHealthChanged` trigger into the actor tick path once the actor is ready.

## Findings

1. `plan_diff` (wire.rs:56) is relay-blind — dead relay with same filter hash as a
   surviving relay will not get a CLOSE; revived relay may not get a REQ.
2. New dead-relay tests assert `current_plan` shape only, not emitted WireFrames —
   the relay-blind diff bug is not caught.
3. `RelayHealthChanged` trigger handling is generic (any inbox trigger recompiles);
   actor tick path not yet wired for A12 in production.
4. API asymmetry: `mark_relay_dead(url: RelayUrl)` by value vs `mark_relay_alive(&RelayUrl)` by ref — polish issue, not a blocker.
5. `set_selection_budget(usize::MAX, usize::MAX)` acceptable for isolation but
   absence of WireFrame assertions means filtering correctness is untested.
