# Codex Review — 5634896

**Subject**: feat(planner): app_relays + drop indexer fallback for content (T134)
**Batch**: HB60-HB66
**Date**: 2026-05-18
**Verdict**: NEEDS-FIX

## Summary

Introduces `UserConfiguredCategory::AppRelay` variant, adds `unroutable_authors:
BTreeSet<Pubkey>` to `CompiledPlan`, and rewires `SubscriptionCompiler::with_relays()`
to accept `app_relays` as an additive lane while removing the indexer fallback for
unknown-mailbox authors in Case A and Case B.

## Rubric

### Correctness

The routing change mostly matches the stated doctrine: Case A/B stop using the indexer
for content, app relays are additive as described, and `unroutable_authors` is a
`BTreeSet` so an author appearing in both `authors` and `addresses` is not
double-counted.

Two correctness problems:

1. **Compile break**: `cargo test -p nmp-core --lib` fails to compile at
   `crates/nmp-core/src/planner/selection.rs:317` because a `CompiledPlan` test
   fixture was not updated with the new `unroutable_authors` field. (99d979d fixes
   this fixture — the issue exists at the 5634896 landing point.)

2. **Wire-diff relay-blindness**: `plan_diff` builds global `BTreeSet<sub_id>` sets,
   not `(relay_url, sub_id)` sets. Adding an app relay for an author already routed
   via NIP-65 creates the same filter hash on a new relay, so the diff skips the new
   REQ. Removing that app relay can also skip the required CLOSE. Fix: scope the diff
   key to `(relay_url, sub_id)`.

### Tests

Six new unit tests cover compiler output directly. Missing:

- Lifecycle/wire behavior when an app relay is added or removed while the same
  filter already exists on a NIP-65 relay.
- Same pubkey in both `authors` and `addresses` with no NIP-65 and no app relays,
  asserting a single entry in `unroutable_authors`.
- JSON deserialization of an older `CompiledPlan` without the `unroutable_authors`
  field (serde default handles JSON but not binary encodings).
- Existing audit test `unknown_author_falls_back_to_indexer` in
  `crates/nmp-testing/tests/m2_subscription_compilation_audit.rs:263` still asserts
  removed indexer fallback semantics and fails against this commit.

### Architecture Fit

`UserConfiguredCategory::AppRelay` fits the existing lane model. `unroutable_authors`
as derived plan metadata fits. Stale docs and compiler module comment still describe
author content fallback to the indexer — conflicts with the new doctrine. Production
`SubscriptionLifecycle` correctly moves to `with_relays()`; remaining
`SubscriptionCompiler::new()` callers are tests that must migrate or be
updated.

### Follow-Ups

- Update `m2_subscription_compilation_audit.rs` stale test.
- Fix wire-diff to scope by `(relay_url, sub_id)`.
- Propagate `unroutable_authors` into the kernel diagnostic path if the
  "author not found on any relay" toast is part of T134's acceptance criteria.
- Update `docs/design/subscription-compilation/compiler.md` to remove
  indexer-fallback references.

## Findings

1. `crates/nmp-core/src/planner/selection.rs:317` fails to compile — missing
   `unroutable_authors` in a `CompiledPlan` initializer.
2. `crates/nmp-testing/tests/m2_subscription_compilation_audit.rs:263` still asserts
   the removed indexer fallback behavior.
3. `plan_diff` in `wire.rs:56-97` diffs by global `sub_id` instead of
   `(relay_url, sub_id)`, so app-relay add/remove can silently skip required
   REQ/CLOSE frames.
4. Stale architecture docs still describe author content fallback to the indexer,
   conflicting with the new doctrine.
