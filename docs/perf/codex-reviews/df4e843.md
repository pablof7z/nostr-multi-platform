had an unambiguous correct behavior:

- **kernel/auth.rs** — added `validate_signed_for` structural guard
  mirroring `nmp_nip42::builder::validate_signed_for` (kind=22242, id len=64,
  sig len=128, challenge tag echo, relay tag presence). Wired into
  `handle_auth_challenge` so a buggy/malicious signer that mutates the kind
  or drops the challenge tag is caught before the kind:22242 hits the wire.
  5 new unit tests pin the guard.
- **ingest/auth_handlers.rs::update_relay_auth_status** now sets
  `changed_since_emit = true` so the diagnostic surface re-emits. Without
  this, a Failed AUTH was non-visible despite `docs/plan/m5-nip42.md` §19
  requiring it. Throughput bounded by actor's ≤60 Hz/view cap (D8).
- **kernel/requests/mod.rs** — added `defer_outbound_silent` for AUTH-pause
  re-defers. The `pending_view_requests` drain → still-paused re-defer loop
  no longer busy-wakes the actor every tick during AUTH-pause windows.
- **kernel/auth_tests.rs Test #5** — renamed-in-name, updated-in-behavior:
  pins that AUTH does NOT directly bump `kernel.rev` (only `make_update`
  does, rate-capped) AND that 10 ticks of auth-pause re-defer do NOT bump
  `changed_since_emit`. Earlier formulation was wrong by the M5 doc.

## Report-only — orchestrator follow-up needed

Two items remain open. Both classified report-only because they require
design calls beyond the fix-in-place mandate:

1. **requests/mod.rs:215 — `Failed` auth pass-through vs fail-closed.**
   AuthGate currently passes Failed-state REQs through (D7 resolution-owner
   contract: the operator owns the resolution path). M5 doc says
   subscriptions stay paused until resolved. Need design call: fail-closed
   (REQs to a Failed relay stay queued indefinitely until user-action
   resolution) vs bounded-drop (current behavior; pass-through with a
   tombstone counter). Recommend filing as an M5 follow-up task with a
   wrong-signer regression test.

2. **auth.rs:1 — kernel duplicates nmp-nip42 protocol module.** Codex notes
   the inlined FSM "duplicates protocol logic". The crate-cycle reasoning is
   documented in the file header (`nmp-nip42 → nmp-core` already exists via
   `substrate::{Signed,Unsigned}Event` + `subs::RelayAuthState`; adding the
   reverse direction creates a Cargo cycle). The category argument
   (NIP-42 is wire-layer like NIP-01 framing, not app-noun like NIP-29) was
   accepted in the merge commit; codex flags it as "resolve before v1". The
   resolution path is the substrate-types extraction (PD-005 in
   `docs/perf/pending-user-decisions.md`); deferring until T59 / M14
   UniFFI scaffolding lands.

3. **auth_tests.rs over 300 LOC soft cap (387).** Codex suggested splitting
   when the behavior fixes land. Precedent:
   `m2_subscription_compilation_audit.rs` at 460 LOC. Deferred — splitting
   now would fragment cohesive tests with no clear seam.
