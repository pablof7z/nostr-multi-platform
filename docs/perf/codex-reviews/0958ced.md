# Codex Review — 0958ced

**Subject**: feat(chirp/onboarding): always-visible NIP-46 section with nostrconnect:// QR + paste
**Batch**: HB60-HB66
**Date**: 2026-05-18
**Verdict**: NEEDS-FIX

## Summary

Removes the `bunkerConnect` mode-gate, makes the NIP-46 section always visible,
adds a collapsible QR code for `nostrconnect://`, an "Open in [App]" shortcut, and
a paste field. On the Rust side, `BunkerBroker::nostrconnect_uri()` generates an
ephemeral keypair and builds the URI. The keypair and secret are not stored.

## Rubric

### Correctness

The paste + Connect flow still works — existing `start_handshake()` is unchanged.
The QR / open-app NIP-46 client-initiated path is wired only for URI generation:

1. **Incomplete handshake** (HIGH): `nostrconnect_uri()` generates a client key and
   secret, then discards both. The broker never subscribes on the client pubkey,
   stores the secret, validates the returned secret from the bunker's `connect`
   response, or discovers the remote signer pubkey. Existing `start_handshake()`
   handles pasted `bunker://` URIs only and generates a separate local key. Per
   NIP-46, the client-initiated flow requires the client to listen for a relay
   response to its pubkey and validate the secret before calling `get_public_key`.

2. **Broken callback** (MEDIUM): `OnboardingView` appends `callback=chirp://nip46`
   to the open-app URL, but `.onOpenURL` in `ChirpApp.swift` only acts when the URL
   contains `bunker_uri` or `uri` query params. A `nostrconnect://` callback that
   returns without those params is a no-op.

### Tests

Seven existing `nmp-signer-broker` tests pass, but none cover:
- `nostrconnect_uri()` field shape, required params, or percent-encoding.
- Secret freshness (each call produces a new secret).
- FFI memory ownership (`nmp_broker_free_string` caller contract).
- A fake-relay client-initiated handshake round-trip.

The iOS UI test bypasses onboarding entirely via `NMP_TEST_NSEC`, so the new screen
can regress silently.

### Architecture Fit

Keeping signer protocol state in `nmp-signer-broker` is correct per D0. The half-done
shape (URI generated, session not held) is the core problem. Swift correctly
displays and opens the token; the broker must own session state and handshake
completion.

`OnboardingView.swift` is now 362 LOC, exceeding the 300-line soft limit. The remote
signer card should be extracted to its own view.

### Follow-Ups

- Implement stored client-initiated session state in `BunkerBroker`: retain
  `(ephemeral_keypair, secret)`, subscribe on the client pubkey, validate the
  secret in the bunker's response, then call `get_public_key`.
- If Phase 2 is intentionally deferred, change UI copy so the QR/open-app path is
  not presented as a working "scan with any NIP-46 signer app" flow.
- Add unit tests for `nostrconnect_uri()` fields and a fake-relay round-trip.
- Split `OnboardingView.swift` to stay under 300 LOC.

## Findings

1. HIGH: `nostrconnect_uri()` generates a client key and secret then drops both;
   the broker never subscribes, stores the secret, validates the response, or
   completes the NIP-46 client-initiated handshake.
2. MEDIUM: the open-app callback path is not a working substitute for the missing
   relay flow — `ChirpApp.swift` `.onOpenURL` is a no-op for `nostrconnect://`
   callbacks lacking `bunker_uri`/`uri` params.
3. MEDIUM: no unit tests for `nostrconnect_uri()` field shape, percent-encoding,
   secret freshness, or FFI ownership; iOS UI tests bypass onboarding entirely.
