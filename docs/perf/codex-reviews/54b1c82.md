### Items NOT addressed (REPORT-class, follow-up needed)

1. **actor_queue_depth not populated** — `kernel/update.rs:68` hardcodes `actor_queue_depth: 0`. The G-S4 gate is now present in S4 but always passes trivially. Follow-up: wire `std::sync::mpsc` channel length (or switch to `crossbeam`) to `Metrics::actor_queue_depth` in the actor loop. This is a multi-file change touching actor/mod.rs + kernel/update.rs + potentially the channel type.

2. **S3 gates not empirically validated** — S3 with 100k Schnorr-signed events runs for ~3-8s signing + 10s settle + 2s burst = ~15s total. Gates were not re-run as part of this commit (no CI/CD with the `ffi-stress` binary in automated testing). The thresholds are set to the same values as before; if Schnorr verify adds more per-emit time, `callback_p99_ms <= 20ms` or `apply_us_p99 <= 16ms` may need to be raised. Follow-up: run `cargo run -p nmp-testing --bin ffi-stress -- s3 --honest` and update `docs/perf/m10.5-results.md`.

3. **G-S4 stale-rev filter proxy is weak** — The `stalls_with_backlog >= stalls_injected` gate is a harness-side proxy for the spec's "stale-rev filter drops >= 1 per stall". In the current harness (Configure-not-Start mode, 4 Hz), emits only fire when configure() is called, so stall windows often have no backlog emits (the callback blocks before new emits can arrive). The gate may fail in this mode. Follow-up: verify stalls_with_backlog semantics when running with real relay workers (`--running` flag if that exists).
