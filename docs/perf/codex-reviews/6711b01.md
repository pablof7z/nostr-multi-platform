# Codex post-merge review - T117 PublishEngine wiring (6711b01)

## Verdict

PARTIAL.

The engine is wired into the live kernel publish path, D0/D3/D6/D7 are mostly
clean, and `accepted_locally` wire shape is preserved. The remaining issues are
REPORT-class: retry scheduling is opportunistic instead of actor-timer driven,
and publish-status reactivity is not yet fully wired for pure ack state changes.

## Findings

- `crates/nmp-core/src/kernel/ingest/mod.rs:42` - Medium - REPORT: publish
  retries only tick on inbound text frames. Quiet relay periods can stall due
  backoff retries indefinitely because the actor idle tick does not call
  `tick_publish_engine_for_now`.
- `crates/nmp-core/src/kernel/ingest/mod.rs:42` / `crates/nmp-core/src/publish/engine.rs:209`
  - Medium - REPORT: high-frequency inbound text frames call `tick` every
  frame; with in-flight publishes, `tick` clones all handles each time. This is
  a D8 hot-path risk until due-retry wakeups are coalesced or actor-timer
  driven.
- `crates/nmp-core/src/kernel/publish_engine.rs:240` - Medium - REPORT:
  `on_ack` can mutate `PublishStatusSnapshot` for OK / terminal failure without
  setting `changed_since_emit` unless retry frames are drained. When the FFI
  publish-status projection lands, pure ack state changes can be invisible to
  snapshot emission unless engine rev changes mark the kernel dirty.
- `crates/nmp-core/src/kernel/publish_engine_tests.rs:3` - Low - REPORT: the
  four tests do exercise the FSM through the kernel seam, not just the shim, but
  they do not cover `publish_signed`, JSON `OK` parsing via `handle_message`,
  actor idle ticking, or live retry dispatch.
- `crates/nmp-core/src/kernel/publish_engine.rs:1` - Info - REPORT:
  `kernel/publish_engine.rs` is 328 LOC, over the 300 LOC soft limit and under
  the 500 LOC hard cap. Not fixed per review instruction.
- `crates/nmp-core/src/kernel/identity_state.rs:39` / `crates/nmp-core/src/kernel/ingest/mod.rs:37`
  / `crates/nmp-core/src/kernel/publish_engine.rs:265` /
  `crates/nmp-core/src/kernel/publish_engine_tests.rs:3` - Low - FIX: stale
  comments omitted new queue statuses or overstated "every frame" / test
  coverage. Fixed in place.

## REPORT-class follow-ups for orchestrator

- Add an actor-owned publish retry pump, likely from the existing 250 ms idle
  loop or a due-deadline timer, and call `resume_publish_engine` on actor/kernel
  startup.
- Avoid per-inbound-frame publish ticking when no retry is due; track next due
  publish retry or let the actor schedule by deadline.
- Make publish-engine snapshot rev changes set `kernel.changed_since_emit` once
  the FFI publish-status projection is wired.
- Add live-path tests for `publish_signed` -> outbound, JSON `OK` through
  `handle_message`, quiet-period retry ticking, and ack-only snapshot emission.
- Keep `PublishQueueEntry::status == "accepted_locally"` for iOS today; migrate
  terminal statuses only with an iOS lockstep change.
