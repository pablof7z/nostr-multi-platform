## Verification matrix (codex output, evidence-cited)

Codex was asked to confirm or refute five claims about the two-commit snapshot.
Results below are codex's findings against `167d4bc + 5c5d417` only (i.e.
*before* `e74247c`/`0849fd2`/`fada22b` landed):

### CLAIM A — `req()` shim routes to bootstrap; live REQ emitters still call `self.req(...)`

**Confirmed at snapshot tip `5c5d417`.** All four sites listed below called
`self.req(...)` which delegates to `req_for_relay(role, role.bootstrap_url(), …)`:

- `crates/nmp-core/src/kernel/requests/mod.rs:87,93,99,105,111,117` —
  `startup_requests` six bootstrap REQs (seed-bootstrap, profile-target,
  target-relays, seed-contacts, seed-profiles, seed-relays).
- `crates/nmp-core/src/kernel/requests/profile.rs:177` (`firehose_requests`),
  `:207` (`profile_claim_request`), `:229/:235/:241` (`author_requests`).
- `crates/nmp-core/src/kernel/requests/thread.rs:133,154` (thread-ids /
  thread-replies hydration).
- `crates/nmp-core/src/kernel/ingest/timeline.rs:209` (`seed-timeline`).

The `req()` function definition at `requests/mod.rs:196-204` proves the
delegation: `self.req_for_relay(role, role.bootstrap_url().to_string(), …)`.
None of these sites threaded a resolved URL.

### CLAIM B — transport layer ignored `relay_url`; pool keyed on `RelayRole`

**Confirmed at snapshot tip `5c5d417`.**

- `crates/nmp-core/src/actor/relay_mgmt.rs:95-109` (`send_outbound`): lookup
  by `message.role` only; `message.relay_url` never read.
- `crates/nmp-core/src/relay_worker.rs:221` (`spawn_relay_worker`): each
  worker dialed `role.url()` (= `role.bootstrap_url()`). The pool held
  exactly two sockets, one per `RelayRole`.
- Publish-side: `publish_cmd.rs:90-100` allocated N `OutboundMessage`s with
  distinct `relay_url` strings, but all with `role = RelayRole::Content`, so
  the transport collapsed all N onto the single Content socket — N duplicate
  sends to the bootstrap URL, no actual fan-out.

This was the load-bearing residual. **Closed by `fada22b`**, which re-keyed
`relay_controls` on `String` (URL), routed `send_outbound` by
`message.relay_url`, and added `ensure_relay_worker` for on-demand per-URL
socket spawning.

### CLAIM C — bootstrap host migration (silent deployment change)

**Confirmed and intentional.** `crates/nmp-core/src/relay.rs:1-2` before T105:
```
pub(crate) const CONTENT_RELAY_URL: &str = "wss://relay.primal.net";
pub(crate) const INDEXER_RELAY_URL: &str = "wss://purplepag.es";
```
After T105:
```
pub(crate) const BOOTSTRAP_DISCOVERY_RELAYS: &[&str] =
    &["wss://relay.damus.io", "wss://nos.lol"];
pub(crate) const CONTENT_RELAY_URL: &str = BOOTSTRAP_DISCOVERY_RELAYS[0]; // damus
pub(crate) const INDEXER_RELAY_URL: &str = BOOTSTRAP_DISCOVERY_RELAYS[1]; // nos.lol
```

Codex's grep for stale fixtures referencing the old hosts found no
production-test pins on `primal.net`/`purplepag.es` outside of an mDocs
research notes file and the design corpus (which discuss them generically,
not as fixtures). The change is intentional — bootstrap relays moved to the
two large-public discovery seeds aligned with `crate::publish::DEFAULT_INDEXER_FALLBACK`.

Operational note: relay-health rows + NIP-42 diagnostics now report the
damus.io/nos.lol hosts as the "Content/Indexer" lane identities until M11
sharpens to per-resolved-URL health tracking. **Not a regression** — the
old hosts were also bootstrap-only.

## Row 1 verdict as of master tip `fada22b`

**Row 1 status: DONE for the follow-feed + author + publish paths.**

What landed across the five T105 commits (`167d4bc → 5c5d417 → e74247c → 0849fd2 → fada22b`):

1. **Read-side resolver** (`kernel/outbox.rs`): `partition_authors_by_write_relays`,
   `author_write_relays`, `recipient_read_relays`, `all_authors_have_relay_lists`,
   `bootstrap_discovery_relays` (5c5d417).
2. **`OutboundMessage::relay_url` field + `req_for_relay()` API + `WireSub::relay_url`**
   so CLOSE routes back to the same socket the REQ went out on (167d4bc).
3. **Live REQ migration** for the three D3-relevant paths (e74247c):
   - `maybe_open_timeline`: follow-feed fans out by per-author NIP-65 write
     relays — one REQ per resolved relay with the authors that relay serves.
   - `author_requests`: kind:1/6 notes target the author's resolved write
     relays (kind:10002 and kind:0 discovery deliberately remain on
     bootstrap — they ARE discovery).
   - `profile_claim_request`: kind:0 follows the author's resolved write
     relays with bootstrap fallback for uncached pubkeys.
4. **A1 recompilation trigger** (`Trigger::Nip65Arrived`, 0849fd2): a kind:10002
   arrival for a follow-feed author marks the timeline for re-planning,
   queues the prior bootstrap-routed seed-timeline-* subs for CLOSE, and
   the next emission re-partitions onto resolved relays. Pinned by three
   integration tests in `kernel/outbox_tests.rs`.
5. **URL-keyed transport pool** (fada22b): `relay_controls: HashMap<String, RelayControl>`
   keyed by resolved URL. `send_outbound` dispatches by `message.relay_url`.
   Per-author/recipient sockets spawn on-demand via `ensure_relay_worker`.
   Bootstrap seeds dialed on Start so cold-start kind:10002 discovery has a
   socket. CLOSE routed back to the socket each WireSub was opened on.
6. **Publish fan-out**: `publish_cmd.rs:89-100` emits one EVENT per resolved
   write+read relay; each carries its own URL; transport dials each socket.

Discrepancies row 1 (D3 outbox automatic) is therefore CLOSED for the live
follow-feed REQ + publish path — the original §27 row 1 claim that the
"live REQ emitters in kernel/requests/{profile,thread}.rs still route via
RelayRole→hardcoded CONTENT_RELAY_URL/INDEXER_RELAY_URL" is no longer
accurate. Both `requests/profile.rs:223/278` and the transport `send_outbound`
now consume the resolved URL, and no read REQ for a warm-NIP-65 author
leaves on the bootstrap seed.

## Residual REPORT-class findings (not fixed in this review pass)

### R1. `thread.rs` hydration still routes through bootstrap `req()`

- **File:** `crates/nmp-core/src/kernel/requests/thread.rs:133,154`
- **What:** `thread-ids-<seq>` (filter: `{"ids":[…]}`) and
  `thread-replies-<seq>` (filter: `{"kinds":[1,6],"#e":[ids]}`) both call
  `self.req(RelayRole::Content, …)` → bootstrap.
- **Severity:** MEDIUM. These are not strict "outbox" queries in the D3
  sense (they look up events by id / `#e` tag, not by author). However the
  ideal outbox routing would extract the original-event author from the
  ids/`#e` set (resolved via `EventStore::scan_by_ids`) and fan out to
  those authors' write relays so reply threads don't depend on bootstrap
  relays carrying the conversation.
- **Why not in this review:** non-trivial. Requires
  store-lookup-then-author-routing logic and probably falls under M2-thread
  / M6 follow-up rather than T105's "follow feed + publish" scope.
- **Suggested follow-up:** task T-thread-outbox: route thread hydration by
  resolving `#e` ids → original-event authors → `author_write_relays`.

### R2. `firehose_requests` (diagnostic hashtag firehose) still routes through bootstrap

- **File:** `crates/nmp-core/src/kernel/requests/profile.rs:187`
- **What:** `diag-firehose-<seq>` (filter: `{"kinds":[1],"#t":[tag],"limit":500}`)
  uses `self.req(RelayRole::Content, …)` → bootstrap.
- **Severity:** LOW. Hashtag firehoses by definition have no `authors`
  filter — there is no per-author outbox to consume. Bootstrap is the
  correct default; the proper destination is the **active account's READ
  relays** (per ADR direction for read-side hashtag interests). Out of
  scope for D3-as-applied-to-follow-feed.
- **Why not in this review:** it's the read-side companion to outbox
  (inbox routing) which is the next milestone band (M2-extension / M6
  inbox); marked `#[allow(dead_code)]` on `recipient_read_relays` is the
  rails for this.
- **Suggested follow-up:** task T-firehose-inbox: route hashtag firehose
  to the active account's read relays via `recipient_read_relays`.

### R3. `startup_requests` discovery REQs intentionally on bootstrap

- **File:** `crates/nmp-core/src/kernel/requests/mod.rs:87,93,99,105,111,117`
- **What:** seed-bootstrap, profile-target, target-relays, seed-contacts,
  seed-profiles, seed-relays — all six on bootstrap.
- **Severity:** NONE. **Correct as-is.** These are the cold-start
  kind:10002 / kind:0 / kind:3 discovery fetches that BY DEFINITION precede
  any cached NIP-65 list. The bootstrap seed is the right destination per
  D3 ("an empty cached relay list falls back to bootstrap only for
  discovery"). No action needed; documented in the
  `BOOTSTRAP_DISCOVERY_RELAYS` doc comment in `relay.rs:7-23`.

### R4. Stale `publish_cmd.rs` module-doc claim (low priority)

- **File:** `crates/nmp-core/src/kernel/publish_cmd.rs:11-14`
- **What:** The pre-T105 doc note "The socket fan-out is the kernel's
