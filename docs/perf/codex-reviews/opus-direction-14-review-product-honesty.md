# NMP Direction Review #14 — 2026-05-24

**Lens:** product-reality vs. planning-claim. Reviews #1–#13 were architecture-focused (substrate purity, doctrine, FFI surface, codegen, ActorCommand enum). They asked "is the kernel sound?" This review asks "is the *app* shippable?" — a different question that the codebase has not been asked recently.

## What the last 13 reviews missed

Every prior review treated Chirp as a fixture for testing the framework thesis, not as a product a user would download. Consequence: the planning docs steadily ticked off *architectural* exit criteria (V-01 WASM stages, V-04 dual-system, V-05 coverage hook, V-12 file ceilings, 9 of 14 `Section 1` violations) while the *product* accreted half-finished features (Search, NIP-42 with bunker, Notifications-as-Outbox, group member list) that no architecture audit flags because they compile, type-check, and pass doctrine-lint. `plan.md` §TL;DR lists "Marmot/MLS encrypted groups" as "what works on master" — true at the kernel layer, false at the user-experience layer. **The gap is a *naming* gap: the docs and lint enforce architectural truth; nothing enforces product truth.**

## Highest-signal finding — `plan.md` v1 exit criteria do not gate live-relay behavior

`/Users/pablofernandez/Work/nostr-multi-platform/.github/workflows/test.yml:41` runs `cargo test --workspace --exclude nmp-android-ffi --exclude nmp-app-chirp --exclude nmp-desktop` — **without `--ignored`**. Every `real_relay_*` test in `crates/nmp-testing/tests/` (`real_relay_smoke.rs`, `real_relay_connect.rs`, `real_relay_nip42.rs`, `real_relay_outbox.rs`, `real_relay_replan.rs`, `real_relay_soak.rs`) is `#[ignore = "real-relay smoke (run with --ignored)"]`. CI never opens a socket to a relay. The "Damus round-trip kind:1" test in `real_relay_smoke.rs:99` is the most basic possible end-to-end proof and it has not been run on a CI box since it was written. F-02 ("DM cold-start receive-side verification") and F-04 ("Zap E2E round-trip verification") are listed as v1 blockers but the docs already acknowledge "device-level acceptance test against live relays (product QA, not CI-gatable)" (`BACKLOG.md:488`) — i.e. v1 will ship with **zero automated assurance that the publish path, the AUTH path, the outbox routing, NIP-77 negentropy sync, or the gift-wrap inbox actually work against a real relay**. The infrastructure to run these tests already exists; the gap is one nightly workflow with `cargo test -- --ignored` plus a "skip if relay unreachable" disclaimer. **Recommended action:** add `.github/workflows/real-relay-nightly.yml` that runs the `real_relay_*` suite on a cron, posts results to a status badge, and gates v1 on the badge being green for 7 consecutive days. Without this, the "v1 ships when F-02/F-04 acceptance test passes" line in the plan is unfalsifiable — there is no place a passing or failing result is recorded.

## Other findings

1. **`SearchView.swift` is dead code that ships in the binary.** `/Users/pablofernandez/Work/nostr-multi-platform/ios/Chirp/Chirp/Features/SearchView.swift:3` defines `struct SearchView` and is compiled into `Chirp.app` (`Chirp.xcodeproj/project.pbxproj:468`), but `grep` finds zero `SearchView()` call sites in the iOS target — `RootShell.swift:7` explicitly documents "search tab removed (Search deferred to toolbar button on HomeFeed)" but no HomeFeed toolbar button to `SearchView` exists either. Worse, the view is mis-named: it does not search Nostr — it is an "open hex pubkey or event id" form (lines 31–47). A user tapping this expects a search box; they get a hex-input field. **Action:** either delete the file, or wire it back into HomeFeed's toolbar AND rename to "Open by ID."

2. **`NotificationsView` is the publish outbox; there are no notifications.** `Features/NotificationsView.swift:7` rustdoc says "Publish-outbox screen." There is no kind:1 mention notification, zap notification, DM notification, or invite notification screen anywhere in `ios/Chirp/Chirp/Features/`. The Settings/About Roadmap (`SettingsHubView.swift:65`) still tells the user "DMs — Direct messages via NIP-04 / NIP-17" is on the roadmap — yet `ChatsView.swift` is a full NIP-17 inbox already shipping (so the in-app copy is stale). **Action:** rename `NotificationsView` → `PublishOutboxView` (with sibling rename in `+OutboxRow.swift`); strip the stale roadmap text and replace with "Mentions / zaps / replies — Notifications" as the genuinely-missing item.

3. **Marmot/MLS has no group-member list — projection returns `Vec::new()`.** `apps/marmot/nmp-app-marmot/src/view/views.rs:270` — `MemberListView::snapshot` returns `MemberListPayload { members: Vec::new() }` with the comment "Authoritative member set is MDK-side; the service/actor layer fills…". `MarmotGroupChatView.swift` has no UI affordance to view group members. The Invite sheet (`MarmotInviteSheet.swift`) accepts npubs but the user cannot see *who is already in the group*. For an encrypted group product, "who can read my messages" is the single most safety-critical question, and the answer is "the user cannot see." **Action:** wire `MDK::get_members()` into `MemberListView::snapshot`, add a sheet to `MarmotGroupChatView`'s toolbar (`...` menu) showing the member list, and gate v1-A Marmot on this.

4. **`DiagnosticsView` (474 LOC, 8 sections, runtime log, queue depth) ships to users via Settings.** `Features/SettingsHubView.swift:43` exposes Diagnostics under a "Developer" section accessible to every user. It surfaces internal facts (Update Seq, Queue Depth, last decode µs, Logical Interests with refcounts, Wire Subscription rows) that no end user can act on, and it is the largest single feature file (474 LOC) — bigger than `MarmotGroupChatView` (246) or `MarmotGroupsView` (354). **Action:** gate Diagnostics behind a debug build flag or a 5-tap-version-row easter egg; replace the user-facing Developer section with a "Connection health" summary (already half-built as `summarySection` of `NotificationsView`).

5. **DM relative-time formatting is in Swift — D6/D0 leak.** `Features/DmListView.swift:284` defines `func dmRelativeTime(_ unixSecs: UInt64) -> String` and `DmConversationView.swift:161` and `DmListView.swift:127` call it. The thin-shell rule says "every relative-time string" is Rust-owned (the Diagnostics file even cites this rule). DM messages bypass it. **Action:** add `createdAtDisplay: String` to `DmMessage` in `nmp-nip17/src/inbox.rs`, delete `dmRelativeTime`.

6. **`PublishOutcome::FailedAfterRetries` writes a `RecentFailure` row but does not fire `set_last_error_toast`.** `publish/state.rs:172` defines the terminal failure variant; `publish/engine/helpers.rs:159` pushes a `RecentFailure` into the publish view; `publish/engine/helpers.rs:274` settles the action ledger with `PublishOutcome::FailedAfterRetries`. The only `set_last_error_toast` call paths from the publish/sign chain are: sign-step failures (referenced in `publish/engine.rs:768` rustdoc, fired by `dispatch.rs:858` and `actor/mod.rs:1300`), broker timeouts (`actor/mod.rs:1402`), and relay-management ops (`actor/commands/relays.rs:151`). No settle-time toast for a publish that fails on every relay. The user sees a red row only if they happen to open the Outbox sheet; otherwise the kind:1 just silently never reaches the network. For a social-app v1 this is a P0 product bug. **Action:** in `publish/engine.rs` settle path, when `PublishOutcome::FailedAfterRetries` resolves, push `set_last_error_toast(Some("Couldn't reach any relay — your post is in the Outbox"))`.

7. **`SettingsHubView.swift:65` Roadmap text is the only place a user sees the product narrative — and it's wrong.** "DMs — Direct messages via NIP-04 / NIP-17" → DMs *ship now* via NIP-17 (Chats tab, `ChatsView.swift`). "Wallet — Lightning wallet integration" → Wallet tab ships. "Push — Real-time push notifications" → genuinely missing but no v1 blocker tracks it. **Action:** delete the Roadmap disclosure or replace with a 3-line current-state summary; the doc-source-of-truth is `docs/plan.md` and that should be what's in the binary, not a separately-drifting list.

## What to kill or deprioritize

- **`SearchView.swift`** — dead code in shipped binary. Delete or wire.
- **Marmot in v1-A** — `MemberListView` returns empty, no member visibility means the encrypted-groups UX is honestly a tech demo. Either land the member-list projection + UI, or move Marmot to a "Labs" tab and remove it from the main `GroupsView` in v1-A.
- **`SettingsHubView.swift` Roadmap text** — stale, no canonical source. Delete.
- **`NotificationsView` as user-facing entry under "Notifications"** — it's the publish outbox. Rename or hide behind a `paperplane` icon labeled "Outbox" only.
- **`DiagnosticsView` from the user-facing Settings** — move to a hidden debug surface.

## 30-day call

**Stop adding architectural exit criteria; spend the next sprint on a `real-relay-nightly` CI workflow + a product-truth pass on every shipping `Features/*View.swift`** (rename `NotificationsView`, delete `SearchView`, fix `SettingsHubView` Roadmap, surface publish `FailedAfterRetries` as a toast, ship the Marmot member list). The thesis "the framework works" is well-defended at the substrate layer (Notes spike, doctrine-lint, perf gate) — but the product that proves the thesis (Chirp) has shipped-binary dead code, mis-named tabs, stale in-app copy, and a "v1 blocker" list whose acceptance tests live only in someone's head. A product-honesty pass costs days, not weeks, and converts the architectural wins into a shippable v1-A.
