# Architectural Polish Scan — 2026-05-21

Targeted scan for the next-highest-impact D3/D4/D6/D8 violations in the NMP
monorepo, run after PRs #88–#91 (projections-map migration, `AtomicBool` for
`pending_mls_autopublish`, correlation_id threading).

**Outcome: no single fix is worth a PR. The scanned surface is clean.**

## Scan 1 — `Arc<Mutex<simple type>>` (D3-adjacent)

```
grep -rn "Arc<Mutex<bool\|Arc<Mutex<u\|Arc<Mutex<i\|Arc<Mutex<String>" crates/ --include="*.rs"
```

One hit: `crates/nmp-core/src/kernel/auth_test_helpers.rs:22` —
`Arc<Mutex<usize>>` in a test-only call-counter helper. Not a production hot
path; no atomic conversion warranted.

`pending_mls_autopublish` is already `AtomicBool` (PR #90). No remaining
`Arc<Mutex<bool/u*/i*/String>>` in production code.

## Scan 2 — `expect()` / `unwrap()` at public boundaries (D6)

```
grep -rn "\.expect(\|\.unwrap()" crates/nmp-core/src/ffi/ crates/nmp-core/src/kernel/
```

Every hit resolves to one of three benign categories:

- **Test modules** — `#[cfg(test)]` blocks and `*_test*.rs` files (the bulk).
- **`conformance_support.rs`** — gated `cfg(any(test, feature = "test-support"))`,
  never compiled into a production build, never on the FFI surface.
- **`doctrine-allow` annotated** — three production sites, each with a written
  justification:
  - `kernel/status.rs:299` — `RelayRole` enum is fixed; a missing seed is a
    logic bug, not a runtime error.
  - `actor/commands/event_observer.rs:181` and `raw_event_observer.rs:206` —
    one-time drain-thread spawn at process init; FFI slot return type cannot
    carry a `Result`; OS thread-spawn failure at startup is unrecoverable.
  - `planner/selection.rs:250` — `max_by` on an iterator proven non-empty
    immediately above.

No unannotated production `unwrap()`/`expect()` in `ffi/` or `kernel/`.

## Scan 3 — Blocking calls inside the actor (D8 / D3)

```
grep -rn "\.lock()\|Mutex::lock\|RwLock::write\|thread::sleep" crates/nmp-core/src/actor/
```

- Every `thread::sleep` is inside a `#[cfg(test)]` module (`relay_mgmt.rs`,
  `tick.rs`, `remote_signer_tests.rs`, `event_observer.rs`,
  `raw_event_observer.rs`). No production sleeps in the actor.
- Every production `.lock()` is **defensive**, not a naive blocking grab:
  `if let Ok(mut guard) = slot.lock()`, `slot.lock().ok().and_then(...)`, or
  `projection_slot.lock().unwrap_or_else(|e| e.into_inner())`. None can panic
  on a poisoned mutex.

### Sanity check: `wallet.rs` `status_slot.lock()` (per memory: "wallet_status is D0")

`actor/commands/wallet.rs` lines 248 / 459 lock `WalletStatusSlot`
(`Arc<Mutex<Option<WalletStatus>>>`). This is **not** a violation: it is the
already-migrated projections-map seam. NIP-47 NWC is an app noun, so wallet
status is deliberately kept out of `KernelSnapshot`; the actor (sole writer,
D4) writes the slot and a host-registered snapshot projection reads it into
`projections["wallet"]` each tick (D0-correct — the kernel emits, never names a
host noun). The lock is the structurally-required actor→projector handoff, the
inner type is `Option<struct>` (not atomic-able), and access is defensive
(`if let Ok`). Leave as-is.
