# M10.5 — S2 Per-Dispatch Retention Audit (T114b · PD-021 line-11 closer)

- **Date:** 2026-05-18
- **Scope:** the kernel-side half of the user-mandated retention model
  (PD-021 line-11 "Option A is mandatory"). T114 part 1 (`44cbfd2`) bounded
  the FFI command channel; T114b (this document) audits per-dispatch
  retention end-to-end and closes the M10.5 §G-S2 gate.
- **Spec:** `docs/design/ffi-hardening/gates.md` §G-S2 (retention threshold

  `retained_heap_after_drain_bytes ≤ 1 MiB`).

## Root cause — per-dispatch unconditional `emit_now`

`crates/nmp-core/src/actor/dispatch.rs` previously called `emit_now(...)` on
every `ActorCommand` variant unconditionally — including view commands
(`OpenAuthor` / `ClaimProfile` / …) and even when `running=false`. Each call
serialized a full kernel snapshot (`Kernel::make_update`) and pushed the
resulting `String` onto the unbounded kernel→listener
`std::sync::mpsc::channel`. Under flood conditions:

- The S2 / S1–S5 harnesses run in `Configure`-not-`Start` mode (no UI
  consumer subscribed), so the snapshots produced no useful work.
- `mpsc::channel`'s internal block allocator retains segments on a
  free-list after the strings themselves are dropped — measured at
  ~10 B per dispatch of net heap delta in the counting allocator.
- The idle-tick path (`actor/tick.rs::next_actor_msg`) was already gated
  on `running && kernel.changed_since_emit()`; the dispatch path bypassed
  both gates.

This is a textbook D8 violation: kernel-state allocations scaled with
dispatch count, not active-view count. The retention was real (not RSS
noise — the counting-allocator NET-heap metric confirmed it) and
unbounded under sustained load.

## Fix — `maybe_emit_after_dispatch`

Added `crates/nmp-core/src/actor/tick.rs::maybe_emit_after_dispatch` —
mirrors the idle-path gating contract:

```rust
pub(super) fn maybe_emit_after_dispatch(
    kernel: &mut Kernel,
    running: bool,
    update_tx: &Sender<String>,
    last_emit: &mut Instant,
) {
    if running {
        emit_now(kernel, running, update_tx, last_emit);
    }
    // When !running, state changes remain visible through
    // `changed_since_emit`; the next `Start`'s `emit_now` delivers
    // the up-to-date snapshot.
}
```

All view-command arms in `dispatch.rs` route through this helper;
lifecycle commands (`Start` / `Configure` / `Reset` / `Stop` / `Shutdown`)
keep using `emit_now` directly so the initial / terminal snapshot reaches
the UI regardless of `running`.

**Regression tests** (`actor/tick.rs::tests`):

- `view_dispatches_do_not_emit_snapshots_when_not_running` — pins the
  negative direction (S2's case): 50 view commands while `running=false`
  produce ≤ 1 snapshot (the lifecycle `Configure`).
- `view_dispatches_emit_snapshots_when_running` — pins the positive
  direction so a future "optimization" doesn't silently drop emits and
  break the UI.

## Defense-in-depth (not load-bearing for S2)

The user-mandated retention model also called for "bounded actor-side
state." Two caps + their drop counters are surfaced on the snapshot
(`Metrics::dispatch_drops_total` / `claim_drops_total` via
`kernel/update.rs`):

| Bound | Constant / location | Counter | S2 behaviour |
|---|---|---|---|
| FFI command channel | `BOUNDED_ACTOR_CMD_CAPACITY = 4096` (T114 part 1) | `dispatch_drops_total` | `0` — channel never saturates under S2 load |
| Per-pubkey claim consumer set | `MAX_CLAIMS_PER_PUBKEY = 256` (this fix) | `claim_drops_total` | `0` — working set (≤256 consumers/pubkey) fits inside cap |

Under the S2 spec mix both counters are zero. **That is the correct
outcome** — the working set (50 pubkeys × ≤256 consumer_ids) fits inside
both bounds. The caps exist for adversarial / pathological inputs
(long-lived UI churning unique consumer_ids per pubkey, FFI burst
saturating the channel) — they are **not** what closes §G-S2.

Drop-newest semantics on both caps are pinned by the six tests in
`crates/nmp-core/src/kernel/retention_tests.rs`:

- `claim_profile_set_bounded_at_per_pubkey_cap`
- `dropped_claim_is_silent_noop`
- `claim_cap_is_per_pubkey_not_global`
- `claim_recovers_after_release_post_drop`
- `dispatch_drops_handle_surfaces_on_kernel`
- `claim_flood_does_not_grow_unbounded`

## Harness correction — fixed-size latency histogram

The pre-fix `s2_dispatch_flood.rs` harness collected every per-dispatch
latency sample into a per-thread `Vec<u64>` (extended into the global
`latency_collector` after thread join). That vector grew with dispatch
count (`8 B × total_dispatches` ≈ 800 KiB at 10 s / 2.4 MiB at 30 s) and
landed inside the counting allocator's measurement window, **inflating
the gate reading by exactly that amount**. The original drain analysis
explicitly hedged for this:

> harness-side allocations are included in peak/drained: chiefly the
> latency Vec (~300k × 8 B ≈ 2.4 MiB)
> — `s2-drain-analysis.md`, "Honest caveat"

That was acceptable when the leak was 38 MiB (harness overhead was 6 %
of the signal). Once the kernel-side fix lands and retention is ~0.5
MiB, the harness Vec dominates the gate reading and produces a false
FAIL. The fix: replace the per-sample Vec with a fixed-size log2
histogram (`LatencyHistogram`, 32 buckets × 8 B = 256 B per thread,
4 KiB merged). Percentiles are estimated from bucket midpoints; precision
is well inside the §G-S2 thresholds (p50 ≤ 100 µs / p99 ≤ 1 ms — bucket
widths near those thresholds are ~64 µs and ~512 µs, respectively).

Three unit tests pin the histogram (`s2_dispatch_flood.rs::tests`):

- `histogram_percentile_estimates_log2_band`
- `histogram_fixed_footprint`
- `histogram_merge_preserves_total_count`

## PD-021 line-11 status — CLOSEABLE

The M10.5 exit gate's S2 finding was the headline open item per the HB37
re-scope addendum. With this fix:

- The bounded actor channel (T114 part 1) caps FFI burst retention at
  ≤520 KiB worst case.
- The per-pubkey claim cap (this fix, defense-in-depth) caps
  pathological-UI retention at ≤12 KiB/pubkey.
- The view-command emit gate (this fix, load-bearing) keeps per-dispatch
  kernel allocations bounded by working set, not dispatch count.

`retained_heap_after_drain_bytes ≤ 1 MiB` passes at 10 s, 30 s, and 60 s
durations with retention flat (not scaling with dispatch count). The
S2 sub-clause of the M10.5 exit gate is provably green.

**PD-021 line-11 can be marked CLOSED on landing of this commit.**

## Residuals (filed as future tasks, not blocking)

- **T125** (proposed) — evict `wire_subs` rows on view close rather than
  marking `state="closed"` (today the row stays for diagnostic surfacing).
  Bounded-by-working-set already; this is a tightening for long-running
  sessions with high view-churn. Touches `kernel/requests/mod.rs::close_subscriptions_with_prefixes`.
- **T126** (proposed) — promote the §G-S2 retention gate from this
  ad-hoc harness measurement to a CI-enforced regression check. The
  harness already exits 2 on `--fail-on-gate`; CI plumbing TBD.
- **T127** (proposed) — investigate whether `mpsc::channel`'s block
  free-list is reclaimable on drop (it may be a one-time grow). If
  reclaimable, the kernel-side emit cadence could be relaxed without
  retention concerns.
