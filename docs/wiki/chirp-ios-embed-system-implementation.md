# Chirp iOS Embed System — Implementation and Architecture

> The embed/kind-registry system (PR #795) was implemented as pure Swift using existing C FFI symbols — zero new FFI, 12 agents across 5 phases, 8 new files. Quoted events now render with native SwiftUI instead of collapsed placeholders.

## Overview

The embed/kind-registry system enables Chirp iOS to render quoted events (kind:1 notes, kind:30023 articles, kind:9802 highlights) with native SwiftUI components instead of collapsed placeholder cards. The implementation was pure Swift — no Rust changes, no new FFI symbols — using C FFI symbols already present in NmpCore.h and projection data the kernel already emits. PR #795 delivered this across 12 agents, 8 new files, in a single PR. [^38935-15]

## Not Blocked on Rust

The embed system was initially characterized as blocked on Rust FlatBuffers codegen. This was incorrect and was retracted. The C FFI symbols nmp_app_claim_event / nmp_app_release_event were already in NmpCore.h. The kernel already emits claimed_events in the snapshot — the gallery reads it successfully from the same libnmp_app_chirp.a that Chirp links against. KernelTypes.generated.swift is generated by a dev-time tool, lives in git, and has precedent for manual field additions. Adding ClaimedEventDto + claimedEvents to SnapshotProjections is a Swift-only edit — no Rust change needed. [^38935-16]


The initial assessment that the embed system was "blocked on Rust" was incorrect and was subsequently retracted. Before claiming a feature is blocked on upstream changes, verify that the C FFI symbols exist in NmpCore.h and that the kernel already emits the relevant projection data. In this case, nmp_app_claim_event / nmp_app_release_event were already present in NmpCore.h, the kernel already emitted claimed_events in the snapshot, and KernelTypes.generated.swift had precedent for manual field additions — no Rust change was needed. The "blocked" claim was investigated and disproven when the user challenged it. [^38935-33]
## Architecture Compliance

The embed plan was verified against all project doctrines before implementation: D8 (no polling) — consumes kernel-pushed snapshots reactively, claim/release are edge-triggered signals; ADR-0025 (no bespoke pull symbols) — uses the existing projection registry seam, mints zero new FFI symbols; One-way principle — one mechanism for projections, the plan uses it; ADR-0037 (typed transport) — reads typed Swift structs from SnapshotProjections, not raw JSON; Component-owned reactivity — components signal their own data requirements via claim/release, kernel never pre-fetches. [^38935-17]

## Implementation Phases

The implementation was organized into 5 phases with 12 agents total. Phase 1 (Foundation, 3 parallel): EmbedKindProjection.swift (projection data types), ClaimedEventDto added to SnapshotProjections (typed FlatBuffers decoding), KernelBridge claim/release + EventClaimSinkProtocol definition. Phase 2 (Bridge + Registry, 3 parallel): EmbedHost.swift (reads claimedEvents from pushed snapshot, resolves envelopes), NostrKindRegistry.swift (kind → renderer dispatch table), EmbedChromeContainer.swift (depth-graded chrome view). Phase 3 (Views, 2 parallel): EmbeddedEvent.swift (claim/release lifecycle view), ArticleEmbed.swift + HighlightEmbed.swift (rich renderers). Phase 4 (Wire, 2 parallel): NostrContentView.swift (embed dispatch path), KernelModel.swift (drives EmbedHost, conforms to EventClaimSink). Phase 5 (Ship): ChirpApp.swift env injection + xcodegen + PR. [^38935-18]

## Deliverables

PR #795 delivered 8 new files: EmbedKindProjection.swift, EmbedHost.swift, NostrKindRegistry.swift, EmbedChromeContainer.swift, EmbeddedEvent.swift, ArticleEmbed.swift, HighlightEmbed.swift, plus modifications to KernelBridge.swift, KernelModel.swift, NostrContentView.swift, ChirpApp.swift, and KernelTypes.generated.swift. Zero new FFI symbols were introduced. After merging, quoted events in the home feed render with native SwiftUI: kind:1 notes as embedded note cards, kind:30023 articles with hero images and title/summary, kind:9802 highlights with pull-quote styling. [^38935-19]

## Related Build Breaks (PR #789 Regressions)

PR #789 (which did the bulk of the component adoption) introduced two build-breaking regressions that made the iOS project uncompilable: (1) a duplicate NostrProfileHost protocol declaration — NostrProfileHost.swift was added as a new file but the same definition was not removed from ProfileWire.swift, causing an invalid redeclaration compile error; (2) ChirpAvatar.init was updated to require a new pubkey: String parameter but all 11 call sites were left unpatched, causing missing argument for parameter 'pubkey' in call. PR #794 fixed both: removing the duplicate from ProfileWire.swift and adding the correct pubkey to each ChirpAvatar call site (item.authorPubkey, conversation.peerPubkey, message.senderPubkeyHex, etc.). These regressions matched the known pattern of PRs merged with red CI. [^38935-20]

## xcodegen Requirement

Chirp iOS uses XcodeGen; project.pbxproj is generated from project.yml. Whenever Swift source files are added (as in the embed system implementation), xcodegen generate must be run before opening the PR to ensure generated project files are current. This is a mandatory final step in any iOS PR that adds or removes Swift source files. [^38935-21]
