# NIP-46 Bunker — First-Class v1 Sign-In Capability

## Sign-in with a NIP-46 bunker

NIP-46 bunker is a first-class v1 sign-in capability. Bunker users must be treated as first-class, migrating `&Keys` consumers to `&dyn RemoteSignerHandle`. When you sign in with a bunker account, zaps from that account work by signing the zap request via NIP-46, just like any other signature the bunker handles. NIP-46 bunker login is implemented using Option A: a signer-broker thread and interface crate pattern, preserving the D0 doctrinal boundary that forbids nmp-core from importing nmp-signers.

Chirp's OnboardingView must support NIP-46 signer detection, bunker:// URI pasting, and nostrconnect:// QR code generation for remote signing. The NIP-46 section in OnboardingView shows a collapsible QR code with the nostrconnect:// URI, an "Open in [App]" button when a signer is detected, and a bunker:// paste field with clipboard detection and handshake progress. NIP-46 signer app detection uses `UIApplication.shared.canOpenURL` with `LSApplicationQueriesSchemes` entries for `nostrsigner`, `primal`, and `nostrconnect`. The `nostrsigner://` URL scheme is generic (used by Amber, Nostore, and others), so the button label should say "Nostr Signer" rather than "Amber".

A new leaf crate `nmp-signer-iface` defines `Nip46Transport`, `Nip46Rpc`, `SignerError`, and a `RemoteSignerHandle` trait with methods `pubkey_hex()`, `signer_kind()`, `sign_async(unsigned, callback)`, and `deliver_relay_event(event)`.

The `BunkerBroker` lives in the FFI layer (or `nmp-signer-broker`), owns the `Nip46Signer` instances, implements `RemoteSignerHandle`, and communicates with the actor via channels.

The `IdentityRuntime` actor holds remote signers in a `HashMap<IdentityId, Arc<dyn RemoteSignerHandle>>`. `ActorCommand::AddRemoteSigner` still takes `Box<dyn RemoteSignerHandle>` and converts to `Arc<dyn>` on insertion, so the boundary API is unchanged.

The actor implements `Nip46Transport` to publish outbound kind:24133 RPCs through the relay pool.

The publish flow is async-aware: `sign_active` returns `Signed | Pending(request_id)`, and the publish queue tracks pending signs. Bunker async-pending modeling extends the existing `PendingSign` machinery to cover NIP-44 operations rather than creating a new state cluster.

New actor commands `StartBunkerHandshake`, `DeliverNip46Response`, and `AddRemoteSigner` support the NIP-46 handshake flow.

The NIP-46 handshake is modeled as an actor state machine: `SignInBunker(uri)` creates a `PendingBunkerAccount`, sends `connect` + `get_public_key`; the UI sees `status: "connecting"` in the snapshot; and upon response the account is upgraded to active.

The NIP-46 handshake snapshot transitions through states `connecting` → `awaiting_pubkey` → `ready` → `idle`.

A bunker NIP-44 adapter bridges `RemoteSignerHandle` to `SignerForSeal`, converting `nostr::UnsignedEvent` to substrate event types and reusing the bunker's `nip44_encrypt` directly.

## nostrconnect URI generation

The nostrconnect URI is generated by the Rust broker (`nmp_app_nostrconnect_uri` FFI) using a fresh secp256k1 keypair with relay, secret, name=Chirp, and requested permissions. The function returns a heap-allocated C string that must be freed via `nmp_broker_free_string`. [^582fc-9]

## Bunker callback handling

Chirp's `chirp://` URL scheme handles incoming `bunker://` callbacks via `onOpenURL` in `ChirpApp`, extracting the `bunker_uri` query param and routing to `model.signInBunker`. [^582fc-10]

## Known follow-ups

Known follow-up: `BrokerTransport::reconnect_hint` is a no-op; mid-session relay disconnects will cause a 45-second sign timeout.

Known follow-up: The MVP holds only one concurrent bunker session; `start_handshake` cancels any prior session.

Known follow-up: Relays requiring NIP-42 auth will not authenticate a remote signer because the active auth signer is cleared; the fix is to wrap the remote signer behind `AuthSignerFn`.

Known follow-up: PR-E Phase 2 (bunker DMs) must extend `PendingSign` rather than using the OS-thread driver pattern.
