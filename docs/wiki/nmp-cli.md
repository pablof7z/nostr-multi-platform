# NMP CLI Crate & Scaffolding Architecture

## Territory & File Boundaries

The nmp-cli crate and its territory are strictly file-disjoint, modifying only the new crates/nmp-cli/ crate, the additive workspace Cargo.toml member entry, and docs/cli.md. The nmp-cli crate may read crates/nmp-codegen/ as a library dependency but must not modify it. Files and directories ios/**, android/**, crates/nmp-core/**, crates/nmp-content/**, crates/nmp-nip23/**, crates/nmp-testing/**, crates/nmp-codegen/**, docs/builder-guide/**, and docs/perf/m10.5/** must not be touched. [^3afdf-2]


## Commands

The `nmp init <app-name>` command scaffolds a new NMP app. It accepts an optional `--path DIR` flag to specify the target directory. Invalid app names are rejected by the `nmp init` command. The `nmp gen modules` command invokes the existing nmp-codegen pipeline by calling it as a library, delegating to the unmodified nmp-codegen library and matching the legacy binary's flags and defaults exactly. Every CLI command must be documented in docs/cli.md. [^3afdf-3]


The CLI component registry (`nmp-cli`) uses an offline built-in TOML manifest to resolve ~30+ UI components via topological dependency sorting. [^54ae9-12]
## Scaffolded App Structure

The scaffold generated by `nmp init` includes an nmp.toml manifest, a workspace Cargo.toml, and a README.md. The scaffolded app-core crate contains one DomainModule, one ViewModule, and one ActionModule over a generic `EntryRecord`, deliberately avoiding a social-app shape to demonstrate doctrine D0. App nouns must live in the app's `<name>-core` crate, never in `nmp-core`. The scaffold includes an `examples/shell.rs` file as a minimal headless shell stub. The scaffolded skeleton pins nmp-core to the checkout's absolute path so that `cargo check` passes green from any directory, including a tempdir. The codegen-emitted `nmp-app-<name>` crate hardcodes `../../../crates/nmp-core` per ADR-0010, so it only builds inside an nmp checkout; this codegen FFI crate's monorepo requirement is documented in docs/cli.md and the generated README.md. [^3afdf-4]

## Testing

The scaffolded app produced by `nmp init` must compile and pass `nmp gen modules` immediately after creation. The integration test for `nmp init` scaffolds into a tempdir, runs `cargo check --all-targets`, runs `cargo test -p <name>-core`, runs `nmp gen modules`, and asserts `nmp gen modules --check` is deterministic with no drift. Tests must verify that `nmp init` creates a tempdir, `cargo check` passes green, and `nmp gen modules` produces deterministic output. The test helpers module in `crates/nmp-cli/tests/helpers/mod.rs` must be a subdirectory (`helpers/mod.rs`), not a flat file (`helpers.rs`), because a flat file would be compiled as its own test binary by Cargo.

## Build & Conventions

The duplicate `nmp` bin name (versus nmp-codegen's legacy bin) must not break `cargo build --workspace`. Guidance on handling this duplicate bin name collision—using `cargo run -p nmp-cli` or `cargo install --path`—is documented in docs/cli.md. Commit messages must use the `feat(cli):` prefix. Temporary proposal crates are deleted and removed from the workspace after a design is selected. TUI mockup and prototype files are standalone crates that declare [workspace] in their own Cargo.toml to opt out of the root workspace and are isolated from the main NMP crates.
