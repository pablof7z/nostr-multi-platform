## CRITICAL ‚Äî UB / crash risk

- [x] **`catch_unwind` on all C-callback fanout** ‚Äî `crates/nmp-core/src/ffi/event_observer.rs:182`, `raw_event_tap.rs:247`, `lifecycle.rs:84`. Every `(registration.callback)(ctx, ptr)` invocation is a foreign call that can unwind across FFI = UB. Wrap each in `catch_unwind`; on panic emit a toast, do not propagate. The actor-thread `catch_unwind` in `ffi/mod.rs:240` does NOT cover these. (fixed `7a2c556f`)
- [x] D6: `expect()` at `ViewModule` boundary ‚Äî `nmp-nip01/src/view.rs` (fixed `0483fe28`)
- [x] D6: `expect()` at grouper public API ‚Äî `nmp-threading/src/grouper.rs` (fixed `44223c5b`)
- [x] D6: `unwrap()` in marmot FFI + gift-wrap policy in wrong layer ‚Äî `marmot/ffi.rs` (fixed `7150202c`)
- [~] **NIP-46 signing blocks the actor thread** ‚Äî Partially addressed. (a) `REMOTE_SIGN_TIMEOUT` reduced from 45s to 5s (`actor/commands/identity.rs:36`) so the worst-case actor stall is bounded. (b) Non-blocking signing now exists: `sign_active_nonblocking` (`identity.rs:251`) returns the `SignerOp` verbatim, and the publish path parks it via `PendingSign` (`actor/pending_sign.rs`), resolving on later ticks. Used by `publish_unsigned_event`, `publish_note`, `react`, `follow` (`actor/commands/publish.rs:57,203,260,317`). (c) **Still open:** the onboarding paths ‚Äî `create_account`, `publish_initial_follows`, `sign_in_nsec` (`identity.rs`) ‚Äî still call the blocking `sign_active`. These are infrequent (account creation / first-follow) so the residual block is low-impact, but a NIP-46 signer there can still stall the actor for up to 5s.
- [x] **Account-switch production path lacks atomicity** ‚Äî Closed (2026-05-20): the "atomicity gap" does not exist. The dead-code half was already fixed (`crates/nmp-signers/src/identity/active_account_reactor/` deleted, commit `8e87ac55`). The remaining concern ‚Äî `switch_active` (`actor/commands/identity.rs:486`) flipping `identity.active` before `sync_kernel` rebinds the signer, with a relay event observing the new pubkey against the old signer in between ‚Äî was investigated and found to be **structurally impossible** under the single-actor model. The close-before-rebind guarantee is provided by the single-actor invariant plus synchronous dispatch, **not** by reordering primitives: there is no concurrent observer of the gap.
  - **Invariant 1 ‚Äî `Kernel: !Send`.** `kernel/mod.rs:397` carries `_not_send: PhantomData<*const ()>`, making any cross-thread kernel mutation a compile error. Exactly one thread ever mutates kernel/identity state.
  - **Invariant 2 ‚Äî dual-channel actor loop drains commands fully before any relay event.** `actor/mod.rs:443-499` is an inner `loop { command_rx.try_recv() }` that drains *all* pending commands; only after it breaks does the outer loop reach `relay_rx.recv_timeout(...)` (`actor/mod.rs:504`). A command and a relay event can never be processed in the same outer iteration ‚Äî relay-event interleaving *within* a command is impossible.
  - **Invariant 3 ‚Äî `dispatch_command` is synchronous.** No `.await`, no yield point. `switch_active` runs flip-`active` ‚Üí `sync_kernel` (rebind auth signer) ‚Üí `reconcile_follow_feed_after_identity_change` ‚Üí `active_account_bootstrap_requests` ‚Üí `retarget_timeline` to completion, then returns, before the loop touches `relay_rx`.
  - **Logical ordering is already correct anyway.** Inside `switch_active`, `sync_kernel` rebinds the auth signer *before* `reconcile_follow_feed_after_identity_change` withdraws the prior account's follow interests (next `drain_tick` emits the CLOSE diff) and *before* `active_account_bootstrap_requests` + `retarget_timeline` produce any new outbound REQ. Every new outbound is therefore generated against the already-rebound new signer; there is no point at which the new pubkey is exposed while the old signer is still bound. A `CloseAccountSubs ‚Üí RebindPublishSigner ‚Üí OpenAccountSubs` reorder would change nothing observable.
- [x] ~~**Silenced memory-safety crash at nsec FFI boundary**~~ ‚úÖ Fixed ‚Äî the crash was **mis-localized** to `parse_secret`. The `*const c_char` ‚Üí `CStr` ‚Üí `&str` seam was never the bug: `KernelBridge.signInNsec` uses `withCString` (Swift side), `c_string_argument` copies to an owned `String` via `to_owned()` (`ffi/mod.rs:582`), the actor owns those bytes in `ActorCommand::SignInNsec { secret: String }`, and `parse_secret(&str)` (`actor/commands/identity.rs:628`) only borrows from that owned String ‚Äî no lifetime hazard exists on that path. The real fault was a **use-after-free of the capability-callback context**: `nmpCapabilityCallback` received an `Unmanaged.passUnretained` pointer to a `ChirpCapabilities` object with no strong owner, so the callback fired with a dangling context during sign-in. Fixed in `970f7fdc`: `KernelHandle` now retains the object (`retainedCapabilities`), registers the handler in `init` before `start()`, and `deinit` unregisters the capability callback before releasing the strong ref. The same commit re-enabled keychain restore in `KernelModel.start()` ‚Äî it is **active now** (`KernelModel.swift:118-123`), so cold relaunch restores local sessions. Follow-up `cc273e54` hardened the retrieve path further: `retrieveSecret` returns a `Codable`-decoded 3-state `SecretLookup` enum instead of brittle JSON `contains()` substring matching. Verified on master: both `KernelModel.swift` and `KernelBridge.swift` carry the fix and are clean.

## HIGH ‚Äî reliability / correctness

- [x] ~~**Observer fanout blocks actor thread**~~ ‚úÖ Fixed ‚Äî C-ABI fan-out uses a bounded `sync_channel(1024)` + drain thread in both `event_observer.rs` and `raw_event_tap.rs`; `notify_observers` / `notify_raw_observers` serialize once, `try_send`, and return immediately. Rust trait observers are intentionally synchronous (their contract mandates cheap + non-panicking).
- [x] ~~**`unsafe impl Send/Sync` false justification**~~ ‚úÖ Fixed (comment only) ‚Äî the old comments claimed "Swift serializes on one thread" (false; `KernelHandle` has no queue). Rewrote to honest three-layer invariant: FFI entry points run on Swift main actor; observer callbacks run on the kernel actor thread (cross-thread); `NmpApp::Drop` joins the actor thread before freeing memory, so Rust-trait observers have no UAF window. **Residual (open):** the C-ABI observer drain thread is NOT joined by `nmp_app_free` ‚Äî a future caller that registers a C-ABI observer and retains `*mut NmpApp` in the callback has a genuine UAF window. Chirp/Marmot use the Rust-trait path and are safe. (commit `6fcfd5ca`)
- [x] ~~**`Kernel: !Send` ‚Äî single-writer invariant not type-enforced**~~ ‚úÖ Fixed ‚Äî `Kernel` carries `_not_send: PhantomData<*const ()>` (`kernel/mod.rs:385`), making any cross-thread share a compile error; the D4 single-writer invariant is now type-enforced.
- [x] ~~**Stale backpressure docs**~~ ‚úÖ Fixed ‚Äî updated four doc comments in `kernel/types.rs` and `kernel/mod.rs` to reflect the unbounded dual-channel design; `dispatch_drops_total` now documented as always-zero diagnostic retained for API compatibility (commit `88c52db2`).
- [x] ~~**Per-message `catch_unwind` in actor**~~ ‚úÖ Fixed ‚Äî `handle_relay_event` wrapped in `catch_unwind(AssertUnwindSafe(...))` in `actor/mod.rs`; panics log the payload + surface an error toast + continue the loop. `kernel::status::log` visibility bumped to `pub(crate)` to make the call accessible (commit `66f41075`).
- [x] ~~**ADR-0009 violation: `nmp-core` ‚Üí `nmp-nwc`**~~ ‚úÖ Clarified ‚Äî `actor/commands/wallet.rs` imports `nmp-nwc` (the NMP protocol-modules row of the ADR-0009 layer table) inside the actor. This is intentional kernel-internal protocol integration: NWC kind:23194 request signing and kind:23195 response decoding mutate wallet state, and D4 (single writer per fact) requires that to happen on the actor thread ‚Äî it cannot be extracted into `nmp-nwc` without a second writer. The dependency is feature-gated (`wallet = ["dep:nmp-nwc"]`, kernel compiles `--no-default-features`) and the direction is kernel ‚Üí protocol (D0 compliant). ADR-0009 amended: the "does not contain" bullet now cross-references a new "Wallet feature exception" subsection. (commit `29553be4`)
- [~] **Zeroization of key material** ‚Äî Largely addressed; one residual upstream-blocked gap remains (see below). Fixed: `Nip46Payload.local_secret_hex` and `.secret` are `Zeroizing<String>` (`8ae6c46c`); `BunkerUri.secret` is `Zeroizing<String>` (`bfc8e5b5`); `NwcUri.client_secret_hex` is `Zeroizing<String>` (`c4e8a76c`); `WalletConnection.client_secret_hex` is `Zeroizing<String>` (`f34230a3`); `LocalKeyMaterial::Raw` is now `Raw(Zeroizing<String>)` (`payload.rs:61`); `LocalKeySigner.secret_hex()` returns `Zeroizing<String>` (`local.rs:143`); `LocalKeySigner.password` is explicitly `zeroize`d in a hand-written `Drop`; `active_local_nsec` is `Arc<Mutex<Option<Zeroizing<String>>>>` (`ffi/mod.rs:186`).
  - **`LocalKeySigner.keys: nostr::Keys` ‚Äî partially mitigated, NOT fully closed.** `LocalKeySigner` now also holds `_secret_bytes: Zeroizing<[u8; 32]>`, captured from the secret at construction and wiped on drop (`local.rs`). This guarantees *a* copy of the raw secret is zeroed from freed memory. **Honest scope of what this does NOT do:** `nostr::Keys` keeps the secret in two further places that are unreachable for zeroing ‚Äî its private `secret_key: secp256k1::SecretKey` field (exposed only as `&SecretKey`, no `&mut` accessor) and a cached `key_pair: OnceCell<Keypair>` (`Keypair` embeds the secret too). `secp256k1` 0.29.1 has **no `zeroize` feature** (verified: no `zeroize` dep, no `Zeroize`/`Drop` impl on `SecretKey`; only a manual `non_secure_erase()` we cannot call without `&mut`), and `nostr` 0.44.2 implements neither `Zeroize` nor `Drop` on `Keys`. So those two internal copies remain in freed heap until reused. The new `_secret_bytes` field reduces ‚Äî but does not eliminate ‚Äî recoverable key material. **Full mitigation requires upstream `Zeroize` support in `nostr::Keys` (or a `nostr`/`secp256k1` upgrade that adds it)** ‚Äî tracked as a residual follow-up, not closed by this change.
- [x] ~~**`SignerPayload` derives `Debug` exposing raw secrets**~~ ‚úÖ Fixed ‚Äî `SignerPayload`, `LocalPayload`, `LocalKeyMaterial`, and `Nip46Payload` no longer `#[derive(Debug)]`; each has a hand-written redacting `Debug` impl (`payload.rs:114-157`) that prints `[redacted]` for all secret-bearing fields, so no `{:?}` log or panic message can leak the private key.
- [x] **nsec crosses FFI as plaintext C string** ‚Äî Rust-side exposure window minimized (commit `0a2fe801`). `ActorCommand::SignInNsec` now carries its secret as `zeroize::Zeroizing<String>` instead of a bare `String` (`actor/mod.rs`); `nmp_app_signin_nsec` (`ffi/identity.rs`) wraps the string in `Zeroizing` the instant it is copied out of the C string (`c_string_argument(...).map(zeroize::Zeroizing::new)`), so the Rust-side copy is wiped on drop ‚Äî including the failure path where `send_cmd` never delivers. The dispatch handler (`actor/dispatch.rs`) passes `secret.as_str()` to `commands::sign_in_nsec`, and the desktop bridge (`nmp-desktop/src/bridge.rs`) wraps before constructing the variant; the whole actor-side path now zeroes on drop. **Honest scope ‚Äî what this does NOT do:** the nsec *inevitably* crosses the FFI boundary as bytes when an account is first imported (it MUST be supplied somehow; Swift still passes a `*const c_char` and `cachedSecretKey: String?` still holds plaintext on the Swift side). `Zeroizing` does not eliminate that transit ‚Äî it only minimizes the in-flight Rust-side window and guarantees the copy is wiped on drop. The earlier `active_local_nsec` doc claim "secret never crosses FFI" remains false for every imported account. Full elimination still requires the Swift-side keychain-write-then-capability-read import path so the nsec never enters a kernel FFI argument at all ‚Äî tracked as a residual follow-up, not closed by this commit.
- [x] ~~**Silent tag drop in `LocalKeySigner::sign_now`**~~ ‚úÖ Fixed ‚Äî `sign_now` (`local.rs:158-169`) now uses `.map(|t| Tag::parse(t)).collect::<Result<Vec<_>, _>>()` and returns `SignerError::Backend` on the first malformed tag, matching the actor's `sign_with` hard-fail; no tag is silently dropped.
- [x] ~~**Observer fanout blocks actor thread (duplicate)**~~ ‚úÖ Already fixed ‚Äî same as item above; bounded channel + drain thread already in place in both observer modules.
