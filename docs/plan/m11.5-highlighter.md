# M11.5 — Highlighter Rebuild + `nmp-nip29` Crate

> **Status:** Design pass complete; ready for `docs/plan.md` ladder insertion at M11.5 (between M11 podcast and M12 — the latter is currently deferred per `docs/plan/scope-adjustments-2026-05-18.md`).
> **Date:** 2026-05-18.
> **Companion docs:** `docs/plan.md` §M11 (the template this follows); `docs/research/highlighter/app-survey.md` (the source-tree map); `docs/research/highlighter/feature-inventory.md` (the user-visible surface); `docs/design/nip29-crate.md` + `docs/design/nip29/{routing,kinds,moderation}.md` (the protocol crate).
> **Why M11.5 exists:** The podcast app (M11) proves the kernel boundary for a non-Nostr-shaped product. M11.5 proves the boundary for a Nostr-shaped product that introduces a *new non-trivial protocol* (NIP-29) on top of the existing social stack, by re-architecting an app that already ships a working Rust core. The "Rust port" step in M11 is greenfield; in M11.5 it's *re-architecting Highlighter's existing `app/core/src/` onto NMP's substrate, deleting its hand-rolled actor + subscription pump + outbox in the process*. The deletion is the proof.

## Demo product

A 1:1 rebuild of `/Users/pablofernandez/Work/hl/app` (Highlighter — Rust core + SwiftUI iOS app, 142 Swift files ≈ 39,200 LOC, Rust core 38 files ≈ 19,900 LOC) running on NMP. **UI is pixel-identical** to the reference iOS app; **all business logic** is in Rust extension modules driving the kernel via the substrate (`DomainModule`/`ViewModule`/`ActionModule`); **NIP-29 lives in a dedicated `nmp-nip29` protocol crate** consumed by the per-app generated crate.

**This is the second load-bearing kernel-boundary check.** If the kernel needs even one community / group / highlight noun to make this work, the boundary is wrong and we fix it. Additionally: if `nmp-nip29` needs to grow special-case hooks into the M2 compiler or M2 publish planner beyond the third routing lane defined in `docs/design/nip29/routing.md`, the routing contract is wrong and we fix that.

### Step 1 — `nmp-nip29` protocol crate (the new protocol surface)

Implement the crate per `docs/design/nip29-crate.md`:

- 11 `DomainModule` impls (`Group`, `GroupAdmins`, `GroupMembers`, `GroupRoles`, `GroupChatMessage`, `GroupDiscussion`, `GroupArtifact`, `GroupRepost`, `GroupReaction`, `GroupComment`, `GroupModerationEvent`). The unifying rule: any event with an `["h", group_id]` tag is a group event owned by `nmp-nip29` regardless of kind — so kind:7 (reactions) + kind:1111 (comments) live here when h-tagged, even though their non-`h` variants live in `nmp-nip25`/`nmp-nip22`. Keeps protocol-crate isolation intact (`nmp-nip25` + `nmp-nip22` know nothing about NIP-29).
- 7 `ViewModule` impls (`JoinedGroups`, `GroupHome`, `GroupChat`, `GroupDiscussions`, `GroupArtifacts`, `GroupMembers`, `GroupExplorer`)
- 15 `ActionModule` impls (`CreateGroup`, `JoinRequest`, `LeaveRequest`, `EditMetadata`, `PutUser`, `RemoveUser`, `CreateInvite`, `DeleteEvent`, `DeleteGroup`, `PostChatMessage`, `PostDiscussion`, `PostArtifact`, `ShareEventIntoGroup`, `ReactInGroup`, `CommentInGroup`)
- Full `RelayPinnedInterest` integration with the M2 compiler (`routing.md` §3) — the third routing lane lands here, not as a special-case in the kernel
- Full `h`-tag override integration with the M2 publish planner (`routing.md` §5)
- `RecentGroupEvents` cache + `previous`-tag attachment logic per `moderation.md` §2
- `JoinedHostsCache` per `routing.md` §4.3 (the per-user host-relay registry derived from prior writes)
- Metadata-signer trust model: ship policy C (best-effort) per `moderation.md` §4.3 with B-ready hooks; ADR documents the chosen policy

### Step 2 — `nmp-nip84` protocol crate (NIP-84 highlights)

NIP-84 is the second new protocol crate M11.5 needs. It's simpler than NIP-29 (one kind: 9802, regular author-write routing, no relay-pin), so the bulk of the design lives in the crate itself, not a separate design doc.

- 1 `DomainModule` (`Highlight`)
- 3 `ViewModule` (`UserHighlights`, `ArtifactHighlights` for "highlights on this article/book/podcast", `FollowingHighlights`)
- 2 `ActionModule` (`PublishHighlight`, `DeleteHighlight`)
- Composed action `PublishHighlightAndShareToGroup` lives at the `highlighter-core` level (it sequences `nmp-nip84::PublishHighlight` + `nmp-nip29::ShareEventIntoGroup`)

## Exit gate (kernel boundary)

- **`nmp-core` gains zero group / highlight / community / room / discussion / artifact / read / book / podcast nouns.** No `Group`, `GroupId`, `Membership`, `Highlight`, `Community`, `Room`, `Discussion`, `Artifact`, `Read`, `Book`, `Episode` types added to the kernel. Verified by grep + manual review at the commit.
- **The M2 compiler gains exactly one generic capability (the third routing lane / `RelayPinnedInterest`), not NIP-29-specific code.** Other relay-pinned NIPs that emerge later can use the same lane without further compiler changes. Verified by code review at the M11.5 PR.
- **The M2 publish planner gains exactly one generic rule (the `h`-tag override).** Verified by code review.
- **`nmp-nip29` does not import any other `nmp-nip*` crate.** Cross-crate joins happen at the application level (in `highlighter-core` or in the generated app crate), not by protocol crates importing each other. Verified at the PR.
- **The ≈ 6,000 LOC of Highlighter infrastructure (`client.rs` + `subscriptions.rs` + `nostr_runtime.rs` + `outbox.rs` + `cache.rs` + `events.rs`) is deleted, not ported.** The replacement is provided by `nmp-core`. Verified by absence in the rebuilt tree.

## Exit gate (product fidelity to `hl/app/ios/Highlighter`)

- **UI parity:** side-by-side screenshot of every screen in `hl/app/ios/Highlighter` vs `ios/NmpHighlighter` matches at ≤ 1 px tolerance (font/rendering differences whitelisted explicitly in `docs/perf/m11.5/parity-screenshots.md`).
- **Feature parity:** every user flow named in `feature-inventory.md` §§ 1–3 reproduced as a scripted Sonnet-agent run on `ios/NmpHighlighter`. No "feature dropped" footnotes.
- **Communities tab end-to-end:** browse the room explorer, preview a room, request to join, receive admit, post a chat message, reply to one, post a discussion, comment on one, share a highlight into the room.
- **Create a room end-to-end:** name + about + picture (uploaded via Blossom) + visibility + access, mint invite codes, share via QR.
- **Admin actions end-to-end:** put-user, remove-user, edit-metadata, create-invite (single + batch fan-out), delete-event.
- **Capture flow end-to-end:** capture from URL + PDF + book + podcast + share-extension; publish-and-share-to-community works without surfacing the dual-routing complexity to the user.
- **Onboarding parity:** paste nsec, generate nsec, paste bunker://, emit nostrconnect:// — all succeed against a real signer.

## Exit gate (routing contract proofs)

All 7 routing tests in `docs/design/nip29/routing.md` §8 + all 7 moderation tests in `docs/design/nip29/moderation.md` §7 pass in `nmp-testing`. The combined 14-test audit is the formal exit gate for the protocol-crate half of M11.5.

## Stress + perf gates

- Joined-rooms list with 50 rooms across 5 host relays renders in ≤ 500 ms cold (no cache) and ≤ 50 ms warm.
- Group chat view with 1000 messages scrolls at 60 fps on iPhone 12.
- Posting a chat message with a populated `previous`-tag cache round-trips in ≤ 800 ms on Wi-Fi (publish → relay-ack → ingest of own event).
- `publish_and_share` dual-route (highlight + share-to-group) completes both writes in ≤ 1500 ms total on Wi-Fi.
- Cold-start with cached LMDB data + 50 joined rooms + 100 admin/member events per room + 1000 cached chat messages: app reaches "rendered home tab" in ≤ 1 second.
- Wire-frame audit: zero off-pin REQs for any NIP-29 filter (verified by the test in `routing.md` §8 test 1).

## Cross-cutting flags

- M11.5 starts only after M11 (podcast) exits. The podcast app's exit gate proved the kernel boundary; M11.5 proves it again with a fundamentally different protocol-density profile (Nostr-shaped, protocol-heavy, single-binary social app).
- M5 (NIP-42 auth) must have shipped: every NIP-29 host relay we've tested requires NIP-42, and `nmp-nip29` declares all its interests + publishes as auth-mandatory per `routing.md` §7.
- M2 (subscription compilation + outbox routing) must have shipped: the third routing lane extends a system that only exists post-M2.
- M3 (LMDB persistence) must have shipped: the `RecentGroupEvents` + `JoinedHostsCache` + `ModerationEvent` records all persist; M11.5 does not have to invent storage.
- M6 (signers + write path) must have shipped: every NIP-29 ActionModule requires a working signer; the bunker://-redeem onboarding path Highlighter expects is M6 surface.
- M10.5 (FFI hardening + iOS empirical proof) must have shipped: M11.5 cannot afford to surface FFI bugs on top of the architectural rewrite.
- M11 (podcast app) provides `podcast-core` + the `AudioPlaybackCapability` lifecycle pattern Highlighter's Podcast feature folder reuses.

## Why M11.5 sits between M11 and M12 in the ladder

Per `scope-adjustments-2026-05-18.md`, M9 (DMs) and M12 (wallet) are deferred to post-v1. M11.5 takes the slot the prior M12 occupied because the kernel-boundary proof case is the load-bearing v1 evidence — two app-rebuilds (podcast + Highlighter), one Nostr-shaped and one not, both proving the substrate handles arbitrary domains without growing kernel nouns. After M11.5 the ladder continues to M13 (WoT), M14 (UniFFI), M15 (cross-platform), M16 (CLI), M17 (v1 release).

## What this milestone does NOT ship

- **Android binary** (`feature-inventory.md` listed Android in the Highlighter source tree, but the directory is absent from the checkout and Android lives in M15).
- **Desktop binary** (same — M15).
- **DMs** — deferred to post-v1 per scope-adjustments.
- **Wallet / NIP-57 / NIP-60 / NIP-61** — deferred to post-v1; Highlighter's "zap a highlight" affordance is therefore *also* deferred. The capture flow + room flow are unaffected.
- **Group migration / forking** (per `nip29-crate.md` §7 deferral list).
- **Encrypted groups** — not a NIP-29 feature; out of scope.
- **iOS Notification Service Extension for group push** — M11.5 ships in-app refresh + local notifications via the existing `LocalNotificationCapability` (added in M10); NSE-driven push is a post-M11.5 feature once Apple's APNs setup is firmed up.

The deferrals are honest — they shrink M11.5's surface to the doctrine-load-bearing parts: NIP-29 as a clean protocol crate, the existing Highlighter app re-architected onto NMP, and the ≈ 6,000-LOC infrastructure deletion as the proof.
