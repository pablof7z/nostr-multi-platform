# M10.5 — FFI hardening + iOS empirical proof *(hard gate before M11 starts)*

> Part of the [Build & Validation Plan](../plan.md). Hard gate before additional app proofs are reopened.

**Demo product:** Chirp's M1-M10 showcase surface subjected to a published, exhaustive stress harness on the iOS simulator and a real iPhone 12. The kernel↔FFI↔SwiftUI path is proven, in measured numbers, to be **rock-solid and demonstrably performant** before a single line of the podcast app is written.

**Why this milestone exists separately.** Every milestone M1–M10 has run iOS measurements, but each in service of its own feature. M10.5 is the dedicated *FFI surface* hardening pass — finding and fixing every shape of FFI bug that a non-social-domain consumer (M11 podcast app) would otherwise discover the hard way. This is also the milestone where we delete every shortcut and "TODO: revisit" comment in the FFI layer.

**Scope.**

- **Stress harness** (`crates/nmp-testing/bin/ffi-stress` + Chirp diagnostics / UI tests):
  - Mount/unmount churn: 1000 view-handle wrappers cycled per second for 10 minutes; assert zero leaks (via Instruments leak instrument scripted run).
  - Dispatch flood: 10k `dispatch(...)` calls per second from Swift across multiple threads; assert no dropped messages, no main-thread block > 16 ms.
  - Snapshot pressure: `AppUpdate::FullState` with 100k events forced; measure marshal time, allocations, and that the reconciler stays ≤ 60 Hz via batching.
  - Reconciler back-pressure: deliberately stall the Swift main thread for 250 ms; assert no actor stall, deltas accumulate and replay correctly when the main thread resumes.
  - Reentrancy: dispatch from inside a reconciler callback (a known footgun); assert ordered, deadlock-free.
  - Capability lifecycle storms: start/stop/restart each registered capability 1000 times; assert idempotency per RMP bible.
  - Error-shape exhaustion: every typed FFI error path exercised; assert each one becomes a `toast: Option<String>` state field, never a thrown exception across the boundary (D3).
  - S11 — Memory RSS instrumentation: capture RSS on iPhone 12 via `mach_task_basic_info` during the canonical workflow (cold start + 10-min sustained scroll + post-shutdown delta); baseline published in `docs/perf/m10.5/iphone12-baseline.md`. Gate: RSS ≤ 100 MB (simulator) / 150 MB (iPhone 12) at 100 active views per ADR-0003 working-set. This is the M1 RSS-measurement deferral being honored here (T22).
- **Real-device measurement on iPhone 12** (one full battery of `firehose-bench live` against primal, all 8 scenarios from `docs/design/firehose-bench.md` §3); produces `docs/perf/m10.5/iphone12-baseline.md` with hardware-tagged numbers.
- **Simulator-driven UI test fleet** (parallel Sonnet agents via the `mcp__xcode` and `BrowserAgent`/`QATester` skills) exercising the app from the outside — boot sim, launch app, tap, scroll, swipe, kill-relaunch — capturing screenshots and assertions per scripted scenario. Every M1–M10 user-visible feature gets a UI test; failures block the milestone.
- **Memory + leak audit** with Xcode Instruments (Leaks, Allocations, Time Profiler) on canonical workflows; zero retained-by-cycle leaks; allocations after warmup linear-or-better in active-view count, never in cached-event count.
- **Profile-Guided Optimization sweep** on the kernel hot paths surfaced by Time Profiler; document tradeoffs taken.
- **All M1–M10 perf reports re-run** on the final FFI surface to confirm no regressions.
- **FFI surface documentation audit** in `docs/ffi-surface.md` — every exported type, function, capability trait, and ownership/lifetime invariant called out; reviewed against `RMP-ARCHITECTURE-BIBLE.md` commandments and ADR-0010.
- **Zero open `TODO`/`FIXME`/`XXX`/`unimplemented!()`** in `crates/nmp-core/src/ffi.rs`, `crates/nmp-core/src/actor.rs`, `crates/nmp-core/src/relay.rs`, `crates/nmp-core/src/kernel/**`, and the Chirp bridge sources (`ios/Chirp/Chirp/Bridge/KernelBridge.swift`, `KernelModel.swift`). Hard zero — no deferral escape, no "tracking issue" carve-out. Every pre-existing one is resolved in M10.5. If something genuinely cannot be done in M10.5 because it belongs to a later milestone (e.g. NIP-65 outbox work), then it is not a TODO/FIXME in the scoped files — it lives as a milestone task in `docs/plan.md`, not as a code marker.

**Subsystem deliverables.**

- `crates/nmp-testing/bin/ffi-stress` — new bench binary.
- Chirp diagnostics, `ChirpUITests`, and smoke scenarios — UI and e2e surfaces driven by both XCTest and (where relevant) a scripted Sonnet-agent runner.
- `docs/design/ffi-hardening.md` — design doc enumerating every FFI failure mode and how the harness exercises it.
- `docs/ffi-surface.md` — the canonical FFI surface reference.
- `docs/perf/m10.5/` — measured numbers from simulator, M-series Mac, iPhone 12; plus screenshots from the Sonnet-driven UI runs.

**Exit gate.**

- All stress-harness scenarios pass on simulator and iPhone 12 with the numeric thresholds enumerated in `docs/design/ffi-hardening.md` §exit-gate.
- All M1–M10 perf reports re-run cleanly on the post-M10.5 binaries; no regression > 5 % on any p99 number.
- Instruments-recorded Leaks count = 0 over the 10-minute canonical workflow.
- Every UI-scripted scenario (Sonnet-agent and XCUITest) passes on a fresh boot of the iPhone 16 Pro simulator and on iPhone 12 hardware.
- `docs/ffi-surface.md` reviewed and tagged.
- Doctrine review (D0–D5) signed off on the FFI surface in writing in `docs/perf/m10.5/doctrine-review.md`.

**Runnable artifact.** Same iOS Twitter app, now load-bearing. Report bundle in `docs/perf/m10.5/`.

## Re-scope addendum (2026-05-18)

**Status:** the exit gate above is **re-scoped, not abandoned.** The original gate
was written before the Pulse e2e app existed and assumes artifacts that do not
exist in this repo. This addendum records the user decision, the gate that M10.5
actually closes on, and the items honestly deferred to the Pulse track. See
[PD-021](../perf/pending-user-decisions.md) for the decision log entry.

### Why the original gate is over-specified

- **No complete M1-M10 Chirp feature stack existed at the time.** M2–M9 are kernel substrate
  (subscription compiler, sync, publish, ledger, relay manager), not
  iOS-integrated user-facing features. M10 (Blossom) was deferred. The
  iOS-integrated e2e app is now **Chirp**; the former **NmpStress** FFI stress
  surface and **NmpPulse** smoke surface were merged into it.
- **No iPhone 12 hardware.** The gate quotes hardware-tagged numbers and a full
  `firehose-bench live` battery on a physical iPhone 12. No such device is in
  the loop. The simulator host is an Apple-Silicon Mac; the default simulator is
  **iPhone 17 Pro** (iOS 26.2), not iPhone 16 Pro — both the original gate's
  "iPhone 16 Pro" and "iPhone 12" device names are stale.
- **"All M1–M10 perf reports re-run"** presumes a finished M1–M10 perf suite
  bound to the FFI surface. The bound suite is the `ffi-stress` S1–S5 harness
  (this milestone's own artifact) plus `firehose-bench`/`reactivity-bench`;
  there is no separate "M1–M10 iOS perf report" set to re-run.
- **"Doctrine review (D0–D5)"** is stale wording. Per
  [PD-001](../perf/pending-user-decisions.md) and later doctrine updates, the
  doctrine set is **D0–D10** (D0–D5 policy + D6–D10 substrate invariants), canonical in
  `docs/product-spec/overview-and-dx.md` §1.5 and `docs/product-spec/doctrine.md`.
  The re-scoped review covers all eleven.

### User decision

Finalize the **simulator-provable subset now**; explicitly and honestly **defer**
the hardware / M10-Blossom items into the Pulse e2e validation track
([`docs/builder-guide/e2e-validation-app.md`](../builder-guide/e2e-validation-app.md)).
Do not fake hardware numbers or re-run a perf suite that does not exist. An honest
"closed on the achievable subset, X deferred with evidence" is required over a
green checkmark that is not true.

### Re-scoped exit gate (what M10.5 closes on — all simulator-provable)

1. **`docs/ffi-surface.md`** — canonical FFI surface reference: every exported C
   symbol in `crates/nmp-core/src/ffi.rs` (production ABI vs `test-support`-gated
   split), every boundary-crossing type, the ownership/lifetime/thread-affinity
   and silent-no-op error contract of each, cross-checked against ADR-0010 and
   the RMP bible. Tagged `reviewed 2026-05-18`.
2. **`docs/perf/m10.5/sim-baseline.md`** — full `ffi-stress` S1–S5 run on the
   simulator host with captured per-scenario p50/p95/p99, allocation counts,
   dropped-message counts, and the gate pass/fail table. Real findings reported,
   not papered over.
3. **Instruments Leaks = 0** over the 10-minute canonical Chirp workflow on
   the iPhone 17 Pro simulator. Parsed leak count + screenshots in
   `docs/perf/m10.5/leak-evidence/`.
4. **UI-scripted simulator fleet** — every Chirp user-visible surface
   exercised from the outside (boot, launch, tap, scroll, swipe,
   kill-relaunch); screenshots + per-scenario PASS/FAIL in
   `docs/perf/m10.5/ui-fleet/`.
5. **`docs/perf/m10.5/doctrine-review.md`** — written **D0–D10** signoff on the
   FFI surface, each doctrine backed by a `file:line` or a `sim-baseline.md`
   number, PASS or a logged exception.

### Deferred into the Pulse e2e validation track

These are **not done and not faked**; they move to
[`docs/builder-guide/e2e-validation-app.md`](../builder-guide/e2e-validation-app.md):

- **iPhone-hardware baseline** — all hardware-tagged numbers, the
  `iphone12-baseline.md` artifact, the per-device scaling coefficient, and the
  S11 `mach_task_basic_info` RSS capture on a physical device. Deferred: no
  hardware in the loop.
- **`firehose-bench live` real-device battery** (all 8 scenarios vs primal on
  iPhone 12). Deferred: requires hardware.
- **"Re-run all M1–M10 perf reports"** — re-interpreted: there is no standalone
  M1–M10 iOS perf suite; the FFI-bound suite is `ffi-stress` (item 2 above).
  Any future feature-level iOS perf reports land in the Pulse track as those
  features become iOS-integrated.
- **M10 Blossom UI scenarios** — Blossom was deferred pre-M10.5; its
  user-visible surface does not exist to UI-script. Re-enters scope when Blossom
  lands in the Pulse track.
- **XCUITest target** (`ios/Chirp/ChirpUITests`) — the re-scoped fleet uses
  external mcp-driven UI automation (item 4). A first-class XCUITest target is a
  Pulse-track hardening follow-up.

The original "Exit gate" and "Scope" sections above are retained verbatim as the
historical record of the pre-Pulse intent; this addendum supersedes them for the
purpose of declaring M10.5 closed.
